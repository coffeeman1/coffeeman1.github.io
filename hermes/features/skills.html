스킬 시스템 ​

643+ Skillsagentskills.ioOpen StandardLearning Loop

스킬은 에이전트에게 '방법(how)'을 가르치는 재사용 가능한 절차적 지식 단위입니다. 프롬프트는 한 번의 지시이고, 도구는 단일 기능이지만, 스킬은 여러 도구를 엮은 워크플로우 + 맥락 + 진화하는 노하우를 하나의 자산으로 묶습니다.

스킬이란 무엇인가 ​

Hermes Agent의 스킬은 네 가지 요소가 결합된 실행 가능한 지식 자산입니다.

절차적 지식
작업을 수행하는 구체적인 단계(how-to)와 의사결정 분기
도구 의존성
어떤 도구/MCP 서버/스크립트가 필요한지 선언
매개변수 계약
입력 스키마, 기본값, 검증 규칙
진화 이력
학습 루프가 누적한 개선 기록과 엣지 케이스

스킬 vs 프롬프트 vs 도구

  • 프롬프트는 "이번 한 번 이렇게 해줘"라는 일회성 지시입니다.
  • **도구(Tool)**는 file-read, web-search 같은 원자적 기능 단위입니다.
  • **스킬(Skill)**은 여러 도구와 프롬프트 조각을 하나의 재사용 가능한 절차로 패키징한 것입니다.

"PR을 리뷰하고 피드백을 달아라"는 프롬프트지만, github-pr-review 스킬은 체크아웃 → diff 분석 → 규칙 기반 리뷰 → 인라인 코멘트 작성 → 요약 보고라는 재사용 가능한 워크플로우입니다.

프로젝트별 스킬(<proj>/.hermes/skills/)은 팀이 레포에 커밋하여 공유할 수 있습니다 — 동일 프로젝트에서 일하는 모든 팀원이 같은 스킬 세트를 쓰게 됩니다.

스킬 파일 구조 ​

스킬은 디렉토리 기반 번들입니다. 한 스킬은 한 디렉토리이며, 최소 SKILL.md 하나만 있으면 유효합니다.

~/.hermes/skills/deploy-to-ecr/
├── SKILL.md                # (필수) 스킬 명세 + 본문 프롬프트
├── skill.yaml              # (선택) 구조화 메타데이터
├── prompts/                # (선택) 재사용 프롬프트 조각
│   ├── step1_analyze.md
│   ├── step2_build.md
│   └── error_recovery.md
├── scripts/                # (선택) RPC로 호출 가능한 유틸리티
│   ├── validate.py
│   └── tag_image.sh
├── examples/               # (선택) few-shot 예시 — 점진적 공개로 로드
│   ├── simple.md
│   └── with_vpc.md
├── references/             # (선택) 배경 문서 — 필요 시에만 로드
│   └── ecr_iam_policy.md
└── tests/                  # (선택) 스킬 회귀 테스트
    └── test_happy_path.yaml

디렉토리 규약 (progressive disclosure)

SKILL.md는 항상 시스템 프롬프트에 요약본만 주입됩니다. prompts/, examples/, references/는 에이전트가 필요하다고 판단할 때에만 읽어들이는 lazy-load 경로입니다 — 컨텍스트 윈도우를 낭비하지 않고 방대한 스킬을 보유할 수 있는 핵심 메커니즘입니다.

SKILL.md 스펙 ​

SKILL.md는 YAML frontmatter + Markdown 본문의 하이브리드입니다.

markdown
---
name: deploy-to-ecr
version: 1.2.0
description: |
  Dockerfile을 분석하고 이미지를 빌드하여 AWS ECR에 푸시한다.
  VPC 전용 ECR, 다중 태그, 커스텀 빌드 args를 지원한다.

author: hermes-community
license: MIT
category: Software Dev
tags: [docker, aws, ecr, deployment, cicd]

# ---- 활성화 트리거 ----
triggers:
  keywords: [deploy, ecr, push image, docker publish]
  file_patterns: [Dockerfile, docker-compose.yml]
  intent: deployment

# ---- 도구/서버 의존성 ----
requires:
  tools:
    - code-execution
    - shell-execute
    - file-read
  mcp_servers: []
  binaries: [docker, aws]
  env_vars: [AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY]

# ---- 입력 계약 ----
parameters:
  - name: project_path
    type: path
    description: Dockerfile이 위치한 프로젝트 루트
    required: true
    validate: "must contain Dockerfile"

  - name: ecr_repo
    type: string
    description: ECR 리포지토리 URI (account.dkr.ecr.region.amazonaws.com/repo)
    required: true
    pattern: '^\d+\.dkr\.ecr\.[a-z0-9-]+\.amazonaws\.com/.+$'

  - name: tag
    type: string
    description: 이미지 태그
    default: latest

  - name: vpc_endpoint
    type: boolean
    description: VPC ECR 엔드포인트 사용 여부
    default: false

# ---- 점진적 공개 ----
progressive_disclosure:
  always_load: [SKILL.md]
  load_on_demand:
    - trigger: "vpc_endpoint == true"
      files: [references/vpc_ecr_setup.md]
    - trigger: "error contains 'denied'"
      files: [prompts/error_recovery.md, references/iam_policy.md]

# ---- 안전 정책 ----
safety:
  approval: required           # required | auto | off
  side_effects: [network, credentials, remote_write]
  reversible: false

# ---- 학습 루프 메타 ----
learning:
  usage_count: 47
  success_rate: 0.94
  last_refined: 2026-03-28
  refinements:
    - { date: 2026-02-10, change: "added VPC endpoint support" }
    - { date: 2026-03-28, change: "handle denied error with IAM fix suggestion" }
---

# Deploy to ECR

이 스킬은 Docker 이미지를 빌드하고 AWS ECR에 푸시하는 표준 절차를 수행합니다.

## 목표
`{{project_path}}`의 이미지를 `{{ecr_repo}}:{{tag}}`로 배포한다.

## 절차
1. **사전 검증** — Dockerfile 존재, `docker`·`aws` 바이너리, AWS 자격 증명 확인
2. **빌드** — `docker build --platform linux/amd64 -t {{ecr_repo}}:{{tag}} {{project_path}}`
3. **로그인** — `aws ecr get-login-password | docker login --username AWS --password-stdin ...`
4. **푸시** — `docker push {{ecr_repo}}:{{tag}}`
5. **검증** — `aws ecr describe-images`로 푸시 확인
6. **요약 보고** — 이미지 digest, 크기, 태그 보고

## 의사결정 분기
- `vpc_endpoint=true`이면 `references/vpc_ecr_setup.md`를 로드하여 VPC 엔드포인트 절차 사용
- `denied` 오류 발생 시 `prompts/error_recovery.md` 로드, IAM 정책 제안
- Dockerfile에 `FROM scratch`가 있으면 multi-arch 경고

## 실패 기준 (do-not)
- 자격 증명이 없으면 **절대** 대화형 `docker login` 시도하지 말 것
- `latest` 태그만 푸시하지 말고, 커밋 해시 태그도 함께 밀 것 (규약)

점진적 공개 (Progressive Disclosure) ​

가장 중요하면서도 흔히 오해받는 개념입니다. 643개 스킬을 모두 시스템 프롬프트에 주입하면 컨텍스트가 터집니다. Hermes는 3단계 로딩 전략을 씁니다.

Level 1 · 카탈로그 인덱스
모든 스킬의 name + description만 요약 인덱스로 주입. 평균 3–5KB. 에이전트가 어떤 스킬이 존재하는지만 인지.
Level 2 · SKILL.md 본문
에이전트가 해당 스킬을 선택하면 그 스킬의 SKILL.md 전체를 로드. 절차·분기·매개변수가 시야에 들어옴.
Level 3 · 필요 리소스
실행 중 조건이 충족되면 references/, examples/, prompts/ 파일을 명시적으로 로드. 에러가 나면 error_recovery.md가 들어오는 식.

이것이 작은 컨텍스트로 거대한 지식을 다루는 비결입니다. 스킬 수가 10배로 늘어도 baseline 주입량은 그대로입니다.

스킬 작성 시 LOD를 염두에 둘 것

SKILL.md 본문이 2KB를 넘어가면 references/, examples/로 쪼개세요. 매번 실행되는 절차는 본문에, 가끔 필요한 배경 지식은 지연 로드 경로에.

스킬 활성화 경로 ​

스킬은 세 가지 방식으로 호출됩니다.

1. 슬래시 명령 (명시적) ​

bash
/deploy-to-ecr project_path=. ecr_repo=123456789.dkr.ecr.ap-northeast-2.amazonaws.com/web tag=v1.4.2

명시적이고 예측 가능. CI 파이프라인이나 크론에서 권장.

2. 자연어 라우팅 (암묵적) ​

> "이 프로젝트 Docker 이미지를 ECR에 올려줘"

에이전트가 triggers.keywords와 description을 매칭하여 deploy-to-ecr을 선택. 선택 결과는 턴 로그에 기록되므로 추적 가능.

3. 스킬 체이닝 (합성) ​

다른 스킬의 본문에서 {{> other-skill}} 문법으로 참조. 아래는 release-flow 스킬이 여러 스킬을 엮는 예시입니다.

markdown
## 절차
1. `{{> run-tests}}` 테스트 통과 확인
2. `{{> bump-version}}` CalVer 태그 생성
3. `{{> deploy-to-ecr tag=$version}}` 이미지 배포
4. `{{> notify-slack channel=#releases}}` 알림

체이닝은 **합성 가능성(composability)**이 스킬 시스템의 본질임을 보여줍니다. 작은 스킬을 조립해 상위 스킬을 만드세요.

재사용성 패턴 ​

좋은 스킬은 한 프로젝트용이 아니라 한 부류의 문제용입니다. 재사용성을 높이는 5가지 원칙:

1. 매개변수화 우선 ​

하드코딩된 경로·URL·자격 증명은 항상 매개변수로 뽑으세요.

yaml
# 나쁨 — 한 번 쓰고 버릴 스킬
description: |
  /home/alice/my-repo의 Dockerfile을 빌드한다

# 좋음 — 어떤 레포에도 쓸 수 있는 스킬
parameters:
  - { name: project_path, type: path, required: true }

2. 단일 책임, 작은 표면 ​

한 스킬 = 한 동사. deploy-to-ecr은 배포만, bump-version은 버저닝만. 합성 가능해야 더 많은 맥락에 붙습니다.

3. 조건부 로직은 분기로, 아닌 것은 별도 스킬로 ​

if vpc_endpoint 정도는 같은 스킬의 분기로 처리하되, AWS ECR과 GCP Artifact Registry는 별도 스킬로 분리하세요. 분기가 3단계 이상이면 리팩터 신호입니다.

4. 실패 경로를 문서화 ​

성공 경로만 쓰지 말고 실패 기준과 복구 절차를 error_recovery.md에 넣으세요. 학습 루프가 여기 쌓는 경험이 진짜 가치입니다.

5. 테스트를 동봉 ​

tests/test_happy_path.yaml에 기대 출력과 부작용을 적어 두면, hermes skills test <name>으로 회귀 검증이 가능합니다. 팀 공유 스킬은 필수.

yaml
# tests/test_happy_path.yaml
name: deploy-to-ecr happy path
given:
  params:
    project_path: ./fixtures/sample-dockerfile
    ecr_repo: 111.dkr.ecr.us-east-1.amazonaws.com/test
    tag: ci
  mocks:
    - { cmd: "docker build*", exit: 0 }
    - { cmd: "aws ecr get-login-password*", exit: 0, stdout: "token" }
    - { cmd: "docker push*", exit: 0 }
expect:
  summary_contains: ["pushed", "ci"]
  side_effects: [network, remote_write]

CLI 명령어 ​

bash
# 목록 & 탐색
hermes skills list                     # 로컬 설치된 스킬
hermes skills list --remote            # agentskills.io 검색
hermes skills search "kubernetes"      # 키워드 검색
hermes skills info deploy-to-ecr       # 스킬 상세
hermes skills show deploy-to-ecr --file SKILL.md

# 설치 & 업데이트
hermes skills install <name>           # agentskills.io에서
hermes skills install github:user/repo # GitHub에서 직접
hermes skills install ./local-skill    # 로컬 디렉토리에서
hermes skills update <name>
hermes skills update --all
hermes skills remove <name>

# 작성 & 검증
hermes skills new my-skill             # 템플릿 생성
hermes skills validate ./my-skill      # SKILL.md 스펙 검증
hermes skills test my-skill            # tests/ 디렉토리 실행
hermes skills lint my-skill            # 스타일/누락 필드 검사

# 실행 & 로그
hermes skills run deploy-to-ecr --dry-run
hermes skills run deploy-to-ecr project_path=. tag=v1.0
hermes skills log deploy-to-ecr        # 최근 실행 로그

# 공유
hermes skills publish my-skill         # agentskills.io 업로드
hermes skills unpublish my-skill --version 1.0.0

스킬 자동 생성 (Learning Loop 연동) ​

에이전트가 복잡한 작업을 마치면 다음 과정이 비동기로 돌아갑니다.

  1. 트라젝토리 추출 — 해당 작업의 도구 호출 시퀀스, 매개변수, 의사결정을 수집
  2. 재사용 가치 평가 — 새로움, 반복 가능성, 부작용 안전성 3축 점수
  3. 스킬 초안화 — LLM이 frontmatter와 절차 본문을 생성
  4. 사용자 리뷰(선택) — auto_persist_skills: prompt 설정 시 에이전트가 확인
  5. 저장 & 인덱싱 — ~/.hermes/skills/<name>/에 기록, 카탈로그 인덱스 재빌드
yaml
# ~/.hermes/config.yaml
skills:
  auto_create: true                 # 학습 루프 자동 생성 활성화
  auto_persist_skills: prompt       # prompt | auto | off
  min_novelty_score: 0.6            # 0–1, 낮을수록 자주 생성
  max_user_skills: 500              # 로컬 상한
  learning:
    refine_on_failure: true         # 실패 시 스킬 업데이트
    track_usage: true

안티패턴 (피해야 할 것) ​

하지 말 것

  • 거대한 메가 스킬 — 배포·테스트·알림을 하나에 몰아넣지 마세요. 합성 가능성 파괴
  • 비결정적 텍스트만 있는 스킬 — 도구 호출 없이 "알아서 해줘"식 본문은 프롬프트지 스킬이 아닙니다
  • 자격 증명 하드코딩 — 항상 env_vars 선언과 런타임 치환
  • 문서 없는 실패 모드 — error_recovery.md가 없으면 같은 실수를 반복
  • 버전 고정 없는 의존성 — MCP 서버나 바이너리 버전을 명시하지 않으면 재현 불가
  • 테스트 없는 공유 — tests/가 없는 스킬을 publish하지 말 것

agentskills.io 오픈 표준 ​

스킬 포맷은 벤더 중립 오픈 표준입니다. 다른 호환 에이전트도 동일 스킬을 로드할 수 있습니다.

  • SemVer 버저닝 — 1.2.0 → 메이저 브레이킹, 마이너 기능, 패치 수정
  • 카테고리 분류 — 16개 표준 카테고리로 검색성 확보
  • 의존성 선언 — requires 필드로 다른 스킬·도구·바이너리 명시
  • 서명(옵션) — 게시자 공개키로 스킬 번들 서명
  • 라이선스 메타 — MIT, Apache-2.0, CC-BY-SA 등 선언 필수

표준 스펙은 agentskills.io/spec에서 확인할 수 있으며, 호환 에이전트 간 상호 운용성을 보장합니다.
