실행 환경 (Backends) ​

6 Backends$5 VPS to HPCInfrastructure Agnostic

동일한 에이전트 코드를 여섯 가지 실행 환경에서 돌릴 수 있게 한 것이 Hermes의 중요한 설계 결정 중 하나입니다. 개인 개발 기기에서부터 대학 HPC 클러스터, 엔터프라이즈 서버리스까지 하나의 스택으로 커버합니다.

6종 실행 백엔드 한눈에 ​

백엔드격리지속성비용 모델주요 용도
Local없음항상고정개발, 프로토타이핑
Docker컨테이너볼륨 마운트고정프로덕션, 격리
SSH네트워크원격 서버고정원격 리소스, 팀 공유
Daytona워크스페이스하이버네이션사용량클라우드 개발
Singularity컨테이너HPC 볼륨고정HPC·과학 컴퓨팅
Modal서버리스하이버네이션사용량비용 최적화, 확장성

각 백엔드는 공통 ExecutionBackend 인터페이스를 구현하므로, 에이전트 코드는 어느 것이 뒤에 있는지 알 필요가 없습니다.

Local ​

호스트 시스템에서 직접 실행. 가장 단순하지만 가장 위험합니다.

yaml
execution:
  backend: local
장점
설정 없이 즉시 사용, 최소 지연, 호스트의 모든 리소스 직접 접근
단점
격리 없음 — 파일·프로세스·네트워크 무제한, 파괴적 명령 위험
권장 대상
개인 개발 환경, 학습·실험, 단기 작업
비권장 대상
팀 공유 머신, 프로덕션, 신뢰할 수 없는 입력을 다룰 때

Docker ​

컨테이너 안에서 격리 실행. 프로덕션의 기본 선택입니다.

yaml
execution:
  backend: docker
  docker:
    image: "hermes-sandbox:latest"
    volumes:
      - { src: "./workspace", dst: "/workspace", mode: "rw" }
      - { src: "./readonly", dst: "/readonly", mode: "ro" }
    network: bridge              # bridge | host | none
    memory: "4g"
    cpus: "2.0"
    pids_limit: 256
    read_only_root: true
    security_opt:
      - "no-new-privileges:true"
    cap_drop: [ALL]
    environment:
      - "TZ=Asia/Seoul"

샌드박스 이미지 ​

bash
# Hermes에 포함된 기본 Dockerfile로 빌드
docker build -t hermes-sandbox -f docker/Dockerfile.sandbox .

# 또는 공식 이미지
docker pull ghcr.io/nousresearch/hermes-sandbox:latest

# 커스텀 — 프로젝트 도구 포함
FROM ghcr.io/nousresearch/hermes-sandbox:latest
RUN apt-get update && apt-get install -y postgresql-client awscli
RUN pip install pandas numpy

Docker-in-Docker? ​

에이전트 안에서 docker build 같은 명령을 쓰려면 DinD 혹은 /var/run/docker.sock 마운트가 필요합니다. 보안 관점에서 sock 마운트는 사실상 root 상승이므로 주의하세요. 대안은 Kaniko·Buildah 같은 rootless 빌더.

SSH ​

원격 서버에서 실행. 로컬 기기의 한계를 넘거나 팀 공유 서버를 활용할 때.

yaml
execution:
  backend: ssh
  ssh:
    host: "user@gpu-server.internal"
    port: 22
    key: "~/.ssh/id_ed25519"
    working_dir: "/scratch/hermes"
    env:
      CUDA_VISIBLE_DEVICES: "0,1"
    keepalive_seconds: 60
    timeout_seconds: 120
장점
원격 GPU/CPU 활용, 데이터가 로컬에 남지 않음, 팀 공유 리소스
단점
네트워크 지연, SSH 키 관리 부담, 연결 끊김 대응 필요
팁
connection multiplexing (ControlMaster)으로 지연 최소화

SSH 키 관리 ​

bash
# hermes 전용 키 생성
ssh-keygen -t ed25519 -f ~/.ssh/hermes_ed25519 -C "hermes@$(hostname)"

# 원격 서버에 배포
ssh-copy-id -i ~/.ssh/hermes_ed25519.pub user@gpu-server

# 권한
chmod 600 ~/.ssh/hermes_ed25519

Daytona ​

클라우드 개발 환경. 하이버네이션 기능이 핵심입니다.

bash
pip install -e ".[daytona]"
yaml
execution:
  backend: daytona
  daytona:
    workspace: "hermes-dev"
    server_url: "https://daytona.example.com"
    image: "daytonaio/python:3.11"
    idle_timeout_minutes: 15
    persistent_volume_gb: 50
bash
# 환경 변수
DAYTONA_API_KEY=xxxxx
DAYTONA_SERVER_URL=https://xxxxx

하이버네이션의 의미 ​

[활성 상태]    → 작업 중, 풀 가격
[유휴 감지]    → N분 미사용 시
[하이버네이트] → 메모리 스냅샷, 비용 0 (스토리지만)
[요청 도착]    → 수백 ms 내 재활성화
[작업 재개]    → 파일·프로세스 상태 보존

이것이 "쓰지 않는 시간에 돈이 나가지 않으면서도 즉시 쓸 수 있는" 환경을 만드는 메커니즘입니다. 간헐적 사용(하루 몇 번)이라면 Docker 고정 비용보다 훨씬 저렴합니다.

Singularity ​

HPC 환경을 위한 컨테이너 시스템. 대학·연구소의 공유 클러스터에 적합합니다.

yaml
execution:
  backend: singularity
  singularity:
    image: "hermes-agent.sif"
    bind:
      - "/data:/data:ro"
      - "/scratch:/scratch:rw"
    gpu: true                    # --nv
    no_home: true                # 홈 디렉토리 마운트 차단
    cleanenv: true
bash
# SIF 이미지 빌드 (루트 불필요)
singularity build --fakeroot hermes-agent.sif Singularity.def

# 또는 Docker에서 변환
singularity build hermes-agent.sif docker://ghcr.io/nousresearch/hermes:latest

HPC 환경 특성 ​

루트 권한 없음
Docker 불가, Singularity/Apptainer만
공유 파일 시스템
Lustre/GPFS — 조심해서 쓰기
SLURM 통합
작업 제출은 SLURM, Hermes는 job 안에서 실행
네트워크 제한
외부 LLM API 접근이 차단될 수 있음 — 로컬 모델 또는 프록시

SLURM 배치 예시 ​

bash
#!/bin/bash
#SBATCH --job-name=hermes-analysis
#SBATCH --nodes=1
#SBATCH --gpus=1
#SBATCH --time=4:00:00
#SBATCH --output=hermes-%j.out

singularity exec --nv \
  --bind /scratch/user:/workspace \
  hermes-agent.sif \
  hermes --non-interactive < analysis_prompt.txt

Modal ​

서버리스 컴퓨팅. 완전 관리형 하이버네이션 + GPU 지원.

bash
pip install -e ".[modal]"
yaml
execution:
  backend: modal
  modal:
    app_name: "hermes-agent"
    image: "python:3.11-slim"
    packages: ["httpx", "pandas"]
    gpu: "T4"                    # None | T4 | A10G | A100 | H100
    memory: 4096                  # MB
    cpu: 2.0
    timeout: 300                  # 초
    keep_warm: 0                  # 0이면 순수 서버리스
    secrets: ["openai-key", "aws-creds"]
bash
# 환경 변수
MODAL_TOKEN_ID=xxxxx
MODAL_TOKEN_SECRET=xxxxx

# Modal 시크릿 설정
modal secret create openai-key OPENAI_API_KEY=sk-...

비용 구조 ​

과금 단위
초 단위, 사용한 리소스(CPU·GPU·메모리)만
cold start
일반적으로 1–3초. GPU는 더 길 수 있음
keep_warm
cold start를 줄이려면 N개 인스턴스를 항상 대기시킴 (고정 비용 발생)
동시성
여러 요청을 하나의 컨테이너에서 처리 가능

Modal이 특히 유용한 경우:

  • 간헐적 사용 (하루 10회 미만)
  • GPU 필요하지만 전용 인스턴스는 과함
  • 부하가 탄력적
  • 완전 관리형 선호 (서버 관리 싫음)

백엔드 전환 ​

bash
# 런타임에서 전환
hermes config set execution.backend docker

# 또는 프로파일로
hermes --profile production    # profile에 backend 지정

세션과 메모리는 유지됩니다 — Persistence Layer는 실행 환경과 독립적입니다. 단, 파일 시스템 상태(working directory 내 파일)는 백엔드마다 다르므로 상대 경로 작업은 주의.

시작하려는 상황은?

조합 패턴 ​

한 프로젝트에서 여러 백엔드를 프로파일로 분리해 쓸 수 있습니다.

yaml
# .hermes/context.yaml
profiles:
  local:
    execution: { backend: local }
  
  ci:
    execution:
      backend: docker
      docker: { image: "ci-sandbox:latest", network: none }
  
  prod-debug:
    execution:
      backend: ssh
      ssh: { host: "debug@prod-jumpbox" }
  
  heavy-compute:
    execution:
      backend: modal
      modal: { gpu: "A10G", timeout: 600 }
bash
hermes --profile heavy-compute    # 이 작업만 Modal GPU로

bash

설계 트레이드오프 ​

왜 6종인가

6종의 백엔드를 유지하는 것은 개발·테스트 부담이 큽니다. 그럼에도 이 결정을 유지하는 이유:

  • 개인부터 엔터프라이즈까지 단일 스택으로 커버
  • 연구기관 HPC는 Singularity 외 대안이 없음
  • 비용 민감 사용자에게 Modal/Daytona는 핵심
  • 빅테크의 서버리스 확산과 함께 가치 증가

대가는 유지보수 부담과 백엔드별 엣지 케이스입니다. 테스트 매트릭스가 커지지만, 이 투자가 **"어떤 인프라에도 맞다"**는 구조적 이점을 만듭니다.
