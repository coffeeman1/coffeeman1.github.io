서브에이전트 ​

Parallel ProcessingIsolated ContextDivide & Conquer

서브에이전트는 메인 에이전트가 스폰하는 독립 컨텍스트의 하위 에이전트입니다. 병렬성을 얻거나, 메인 컨텍스트를 오염시키지 않고 깊은 탐색을 수행하거나, 작업을 분할 정복할 때 씁니다.

왜 서브에이전트가 필요한가 ​

에이전트는 단일 컨텍스트 윈도우 안에서 작업합니다. 다음 상황에서 이 제약이 문제가 됩니다.

대량 파일 탐색
100개 파일을 한 번에 읽으면 컨텍스트가 터짐 — 서브에이전트가 각자 읽고 요약만 리턴
병렬화 기회
세 API를 동시에 테스트할 수 있다면 순차 실행할 이유가 없음
실험적 탐색
"세 가지 방법으로 구현해 봐" — 실패한 시도가 메인 컨텍스트를 더럽히지 않음
격리된 도구 세트
서브에이전트에게는 제한된 도구만 줘서 안전하게 실행
비용 통제
서브에이전트를 저렴한 모델로, 메인은 고급 모델로

작동 모델 ​

┌─────────────────────────────┐
│       Main Agent            │
│  모델: claude-3.5-sonnet    │

│독립 ││독립 ││독립 ││독립 │
│컨텍 ││컨텍 ││컨텍 ││컨텍 │

핵심 포인트: 서브에이전트는 메인 에이전트의 "수명을 공유하지 않는" 독립 인스턴스입니다. 작업 완료 후 자동 정리되며, 메인 컨텍스트에는 최종 요약만 전달됩니다.

스폰 API ​

서브에이전트는 spawn-subagent 도구로 생성됩니다.

json
{
  "task": "frontend/ 디렉토리의 모든 React 컴포넌트에서 useState 사용 패턴을 조사하고, 개선 가능한 후보 5개를 요약하라",
  "toolset": "read-only",
  "model": "openrouter/anthropic/claude-3-5-haiku",
  "max_turns": 20,
  "timeout_seconds": 300,
  "inherit": {
    "skills": false,
    "memory": "read-only",
    "profile": true
  },
  "return_format": "structured"
}

파라미터 ​

task
서브에이전트에게 줄 단일 지시문. 명확하고 완결된 문장이어야 함
toolset
제한된 도구셋. read-only, web-research, sandbox-exec 등
model
서브에이전트가 쓸 모델. 기본은 메인과 동일, 비용 절감 시 경량 모델 지정
max_turns
사고-도구 호출 루프 최대 횟수. 무한 루프 방지
timeout_seconds
wall-clock 시간 제한
inherit.skills
메인의 활성 스킬을 물려줄지. 기본 false
inherit.memory
full/read-only/none
inherit.profile
프로파일(프롬프트 레이어)을 물려줄지
return_format
text(자연어)/structured(JSON)/markdown

전형적인 사용 사례 ​

병렬 조사 ​

사용자: "이 프로젝트의 API 3개가 각각 어느 페이지에서 호출되는지 찾아줘"

메인 에이전트:
  spawn-subagent(task="/api/users를 호출하는 모든 곳 찾기", toolset="read-only")
  spawn-subagent(task="/api/orders를 호출하는 모든 곳 찾기", toolset="read-only")
  spawn-subagent(task="/api/products를 호출하는 모든 곳 찾기", toolset="read-only")

각 서브에이전트 결과:
  Sub-1: "users: 4곳 — profile.tsx, settings.tsx, admin/users.tsx, api/auth/me.ts"
  Sub-2: "orders: 3곳 — cart.tsx, checkout.tsx, admin/orders.tsx"
  Sub-3: "products: 6곳 — home.tsx, search.tsx, ..."

메인 에이전트가 통합 리포트 생성 (총 13곳)

분할 정복 리뷰 ​

사용자: "이 모노레포 전체 코드를 보안 관점에서 리뷰해줘"

메인 에이전트: [모듈별 서브에이전트 스폰]
  Sub-1: packages/auth (인증·세션)
  Sub-2: packages/api (입력 검증·SQL)
  Sub-3: packages/ui (XSS·CSP)
  Sub-4: infra/ (시크릿·권한)

각 서브에이전트:
  - 해당 모듈만 스캔
  - 발견한 이슈를 JSON으로 반환
  - 메인 컨텍스트에는 로드하지 않음

메인 에이전트:
  - 네 개의 JSON을 받아 우선순위 종합
  - 중복 제거, 카테고리별 정리
  - 최종 보고서 작성

실험적 구현 비교 ​

사용자: "이 해싱 함수를 3가지 방법으로 구현하고 벤치마크해줘"

메인 에이전트:
  spawn-subagent(task="방법 A: 표준 해시맵으로 구현 + 벤치마크", toolset="sandbox-exec")
  spawn-subagent(task="방법 B: LRU 캐시로 구현 + 벤치마크", toolset="sandbox-exec")
  spawn-subagent(task="방법 C: 블룸 필터 + 해시맵 하이브리드 + 벤치마크", toolset="sandbox-exec")

결과: 각 서브에이전트가 /tmp에 구현 + bench 결과 JSON 반환
메인 에이전트: 비교표 + 추천 작성

안전한 실험 ​

사용자: "이 마이그레이션 스크립트가 실제로 작동하는지 확인해줘"

메인 에이전트 (프로덕션 컨텍스트, 쓰기 권한 있음):
  spawn-subagent(
    task="마이그레이션 스크립트를 docker sandbox에서 실행하고 결과 확인",
    toolset="sandbox-exec",
    inherit={ memory: "none" }      # 메모리도 격리
  )

서브에이전트는 샌드박스 안에서만 작업 → 실제 DB에는 영향 없음

격리 수준 ​

서브에이전트의 격리는 세 층위로 제어됩니다.

1. 컨텍스트 격리 ​

항상 독립된 컨텍스트 윈도우. 메인의 대화 히스토리가 서브에이전트에게 주입되지 않습니다. 이것이 기본 보장입니다.

2. 실행 환경 격리 (선택) ​

json
{
  "task": "...",
  "execution": {
    "backend": "docker",
    "image": "sandbox:latest",
    "network": "none",
    "memory_mb": 1024,
    "cpu_limit": 1.0
  }
}

서브에이전트를 별도 Docker 컨테이너에서 실행 — 파일 시스템 쓰기·네트워크도 격리. 자세한 것은 실행 환경 참조.

3. 권한 격리 ​

json
{
  "toolset": "read-only",
  "security": {
    "approval_required": true,
    "blocked": ["shell-execute", "file-write", "message-send"]
  }
}

결과 종합 패턴 ​

메인 에이전트가 여러 서브에이전트의 결과를 종합하는 방식:

패턴 1 · 단순 병합 ​

python
# 메인 에이전트의 사고 (의사 코드)
results = spawn_parallel([
  Sub(task="A"),
  Sub(task="B"),
  Sub(task="C"),
])
final = merge_reports(results)      # 단순 concat + 중복 제거

패턴 2 · Map-Reduce ​

python
# Map 단계
chunks = split_codebase(by="module")
partial = spawn_parallel([
  Sub(task=f"review {chunk.path}") for chunk in chunks
])

# Reduce 단계
categorized = reduce_by_category(partial)  # 이슈를 카테고리별로
prioritized = rank_by_severity(categorized)
final = generate_report(prioritized)

패턴 3 · 경쟁적 탐색 ​

python
# 같은 문제를 여러 접근으로
candidates = spawn_parallel([
  Sub(task="approach: cache with LRU"),
  Sub(task="approach: cache with TTL"),
  Sub(task="approach: cache with size limit"),
])

# 실제 벤치마크 결과로 선택
best = select_by_metric(candidates, metric="latency_p95")

패턴 4 · 연쇄 위임 ​

python
# 서브에이전트가 또 서브에이전트를 스폰
main spawns sub1
  sub1 spawns sub1.1, sub1.2 for deeper exploration
    sub1.1 returns
    sub1.2 returns
  sub1 synthesizes and returns to main

최대 깊이는 기본 2단계로 제한됩니다(subagents.max_depth).

비용 통제 ​

서브에이전트는 비용을 곱셈으로 늘릴 수 있습니다 — 10개 서브에이전트 = 10배 토큰. 다음 전략으로 관리하세요.

yaml
# ~/.hermes/config.yaml
subagents:
  enabled: true
  max_concurrent: 5              # 동시 실행 상한
  max_depth: 2                   # 재귀 깊이
  default_model: openrouter/anthropic/claude-3-5-haiku  # 기본 경량
  default_timeout_seconds: 180
  default_max_turns: 15
  cost_cap:
    per_task_tokens: 20000       # 단일 서브에이전트 토큰 상한
    per_main_turn_tokens: 100000 # 메인 턴당 전체 서브 토큰 상한
  require_explicit_spawn: false  # true면 사용자 승인 필요

크론 + 서브에이전트 ​

가장 실용적인 조합 중 하나입니다.

사용자: "매일 오전 8시에 우리 5개 마이크로서비스의 헬스체크를 병렬로
       실행하고, 이상 있는 항목만 Slack #ops에 알림"

크론 트리거 → 메인 에이전트 →
  spawn-subagent("auth-service 헬스체크", toolset="network-only")
  spawn-subagent("api-service 헬스체크", toolset="network-only")
  spawn-subagent("worker-service 헬스체크", toolset="network-only")
  spawn-subagent("gateway-service 헬스체크", toolset="network-only")
  spawn-subagent("cdn-service 헬스체크", toolset="network-only")

메인 에이전트:
  - 5개 결과 수집
  - 이상 있는 것만 필터 (e.g. auth, gateway)
  - Slack 메시지 조립: "⚠️ auth: 503, gateway: timeout"
  - message-send("slack", "#ops", ...)

5개 서비스를 순차 확인하면 5×N초, 병렬이면 max(N)초. 비용도 병렬이 유리 — 전체 대화 토큰이 아니라 필요한 요약만 메인에 올라옵니다.

디버깅 ​

bash
# 최근 서브에이전트 목록
hermes subagents recent

# 특정 서브에이전트 상세
hermes subagents show <id>

# 실시간 로그
hermes subagents logs <id> --follow

# 강제 종료
hermes subagents kill <id>

# 전체 정리
hermes subagents cleanup

한계와 함정 ​

서브에이전트를 남용하지 말 것

  • 단순 작업은 도구 호출이면 충분합니다. 서브에이전트는 오버헤드가 있습니다 (새 컨텍스트, 시스템 프롬프트 재로드)
  • 직렬 의존성이 있으면 병렬의 의미가 없음 — A의 결과로 B를 해야 한다면 한 에이전트가 순차로
  • 너무 얕은 작업을 넘기면 도구 호출보다 나을 게 없음
  • 상태 공유가 필요한 작업에는 부적합 — 서브는 메인의 변수를 모름
  • 메인 에이전트가 결과를 해석할 수 있을 만큼 구조화된 출력이 필요 — return_format: structured 권장

설계 원칙 ​

  1. 태스크는 자기 완결적이어야 함 — "이 작업의 나머지를 계속해"는 안 됨
  2. 입력과 출력을 명확히 — 서브에게 필요한 정보만 전달, 반환 형식 지정
  3. 실패를 격리 — 서브 한 개가 실패해도 나머지는 계속
  4. 비용을 인지 — 서브 5개 × 각 20K 토큰 = 100K 토큰 추가 비용
  5. 도구 권한을 좁게 — 필요한 것만, read-only 우선
  6. 타임아웃을 설정 — 무한 루프 가능성

연결 문서

  • 실행 환경 격리 → 실행 환경
  • 크론과의 조합 → 크론 스케줄러
  • 도구 승인 정책 → 보안 체계
