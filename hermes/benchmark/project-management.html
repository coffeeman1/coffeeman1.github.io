CAT-17. 프로젝트 매니지먼트 ​

평가 핵심: 에이전트가 WBS, 리스크 관리, 이해관계자 관리, 회고 등 PM 실무 문서를 체계적으로 작성할 수 있는가?

PM-01. 프로젝트 킥오프 문서 세트 ⭐⭐⭐ ​

시나리오 배경 ​

프로젝트 시작 시 필요한 핵심 문서들을 한 번에 만드는 능력.

프롬프트 ​

아래 프로젝트의 킥오프 문서 세트를 만들어줘.

프로젝트 개요:
- 프로젝트명: "디지털 전환 Phase 1 — 고객 포털 구축"
- 발주: 중견 제조업 M사 (매출 3,000억, 직원 800명)
- 수행: 우리 SI 기업 (PM 포함 10명 투입)
- 기간: 6개월 (2024.04 ~ 2024.09)
- 예산: 5억원
- 목표: 기존 전화/팩스 기반 주문을 온라인 포털로 전환
- 주요 기능: 온라인 주문, 실시간 재고 조회, 배송 추적, 정산 내역
- 이해관계자:
  - M사 측: 디지털혁신팀장(PM 카운터파트), IT팀장, 영업본부장, CEO(스폰서)
  - 우리 측: PM, PL, 개발 5, 디자인 1, QA 1, BA 1

산출물:

1. `pm/project-charter.md` — 프로젝트 헌장
   - 프로젝트 배경 & 목적
   - 범위 (In scope / Out of scope 명확히)
   - 핵심 마일스톤 (6개)
   - 성공 기준 (정량적: 주문 전환율 70%, 정성적: 사용자 만족)
   - 가정 & 제약조건
   - 리스크 초기 식별 (TOP 5)
   - 승인 서명란

2. `pm/wbs.md` — 작업 분해 구조
   - Level 0: 프로젝트 전체
   - Level 1: 페이즈 (분석 → 설계 → 개발 → 테스트 → 이관 → 안정화)
   - Level 2: 각 페이즈별 작업 패키지
   - Level 3: 세부 태스크 (담당자, 기간, 선행 태스크)
   - 총 태스크 수: 60~80개
   - 크리티컬 패스 식별

3. `pm/stakeholder-map.md` — 이해관계자 분석
   - 이해관계자 매핑 (영향력 × 관심도 매트릭스)
   - 각 이해관계자별:
     - 역할, 기대사항, 우려사항
     - 커뮤니케이션 빈도 & 방법
     - 관리 전략 (Manage Closely / Keep Satisfied / Keep Informed / Monitor)
   - 특별 주의: 영업본부장 (현업, 변화 저항 가능성)

4. `pm/communication-plan.md` — 커뮤니케이션 계획
   - 정기 회의: 주간(내부), 격주(고객), 월간(스티어링 커미티)
   - 각 회의: 참석자, 아젠다 템플릿, 산출물
   - 보고서: 주간 보고, 월간 보고, 이슈 보고 형식
   - 에스컬레이션 절차 (이슈 → 리스크 → 변경요청)
   - 비상 연락 체계

파일 경로: 위에 명시된 대로.

평가 포인트 ​

  • [ ] 프로젝트 헌장의 형식과 내용 완성도
  • [ ] In/Out of Scope의 명확성 (모호함 없이)
  • [ ] WBS의 적절한 깊이 (60~80개 태스크)
  • [ ] 크리티컬 패스 식별의 정확성
  • [ ] 이해관계자 매트릭스의 현실성
  • [ ] 영업본부장의 변화 저항 관리 전략
  • [ ] 커뮤니케이션 계획의 체계성

PM-02. 리스크 관리 계획서 ⭐⭐ ​

시나리오 배경 ​

프로젝트 리스크를 식별·분석·대응하는 체계적 문서.

프롬프트 ​

위 "고객 포털 구축" 프로젝트의 리스크 관리 계획서를 만들어줘.

산출물: `pm/risk-management-plan.md`

1. 리스크 관리 프로세스
   - 식별 → 분석 → 대응 계획 → 모니터링
   - 리스크 리뷰 주기 (격주)
   - 리스크 오너 지정 기준

2. 리스크 레지스터 (15개 이상)
   - 각 리스크:
     - ID, 제목, 설명
     - 카테고리: 기술 / 일정 / 비용 / 인력 / 외부 / 범위
     - 발생 확률 (1~5)
     - 영향도 (1~5)
     - 리스크 점수 (확률 × 영향도)
     - 대응 전략: 회피 / 완화 / 전가 / 수용
     - 구체적 대응 방안
     - 리스크 오너
     - 트리거 (이 조건이 되면 리스크 현실화)
     - 현재 상태: Open / Monitoring / Closed

   포함해야 할 리스크 (최소):
   - R01: 핵심 개발자 이탈
   - R02: 고객사 요구사항 변경 (Scope Creep)
   - R03: 레거시 시스템 연동 실패
   - R04: 고객사 데이터 마이그레이션 지연
   - R05: 현업(영업부) 사용자 저항
   - R06: 외부 API 의존 (ERP 연동)
   - R07: 일정 지연 (크리티컬 패스 지연)
   - R08: 보안 취약점 발견 (고객 데이터)
   - + 7개 이상 추가

3. 리스크 히트맵
   - 확률 × 영향도 매트릭스 (5×5)
   - 각 리스크를 매트릭스에 배치
   - 빨강(즉시 대응) / 노랑(주의) / 초록(모니터링) 구분

4. 컨틴전시 계획 (TOP 5 리스크)
   - 리스크가 현실화되면 구체적으로 뭘 할 것인가
   - 예비 예산 / 일정 버퍼 / 대체 인력 계획

5. 변경 관리 프로세스
   - 변경 요청서(CR) 양식
   - 변경 영향도 평가 기준
   - 변경 승인 프로세스 (누가 승인, 기준)
   - Scope Creep 방지 전략

평가 포인트 ​

  • [ ] 15개 리스크의 현실성 (SI 프로젝트 특유의 리스크)
  • [ ] 확률/영향도 평가의 일관성
  • [ ] 대응 전략(회피/완화/전가/수용)의 적절한 선택
  • [ ] 트리거 조건의 구체성
  • [ ] 히트맵의 시각적 명확성
  • [ ] 컨틴전시 계획의 구체성
  • [ ] Scope Creep 방지 전략의 실효성

PM-03. 프로젝트 상태 보고서 ⭐ ​

시나리오 배경 ​

PM이 매주 작성하는 상태 보고서. 빠르게 상황 파악이 되어야 함.

프롬프트 ​

프로젝트 상태 보고서 템플릿과 샘플을 만들어줘.

프로젝트: 위와 동일 (고객 포털 구축, 현재 3개월차)

현재 상황:
- 전체 진척: 45% (계획 50%, -5% 지연)
- 설계 단계 완료, 개발 단계 3주차
- 이슈: ERP 연동 API 스펙이 고객사에서 아직 미제공 → 개발 블로킹
- 리스크: 개발자 1명 건강 이슈 (2주 병가 예정)
- 예산: 집행 2.2억 / 전체 5억 (44%, 정상)
- 마일스톤: "설계 완료" 1주 지연 후 달성, "개발 50%" D-14

1. `pm/weekly-report-template.md` — 주간 보고 템플릿

   구성:
   - 신호등 (전체 상태: 초록/노랑/빨강)
   - 한 줄 요약
   - 핵심 지표 대시보드:
     - 진척률 (계획 vs 실적)
     - 일정 (마일스톤 달성 현황)
     - 예산 (집행 현황)
     - 품질 (버그 수, 테스트 진척)
     - 리소스 (투입 현황)
   - 금주 완료 사항 (Done)
   - 차주 계획 (To Do)
   - 이슈 (미해결, 신규)
   - 리스크 (변동 사항)
   - 의사결정 필요 사항 (있을 경우)
   - 고객 커뮤니케이션 요약

2. `pm/weekly-report-sample.md` — 3개월차 샘플 보고서
   - 위 템플릿에 현재 상황 데이터를 채운 실제 보고서
   - 신호등: 노랑 (ERP API 블로킹 이슈로)
   - 이슈 상세 설명 + 요청 사항 (고객사 API 스펙 4/15까지 필요)
   - 개발자 병가 대응 계획

3. `pm/steering-committee-report.md` — 월간 경영진 보고
   - 주간 보고와 다른 점: 
     - 더 요약적 (1~2페이지)
     - 의사결정 사항에 집중
     - 비즈니스 임팩트 관점
   - 샘플 작성

평가 포인트 ​

  • [ ] 신호등 체계의 명확한 기준
  • [ ] 한 줄 요약의 핵심 전달력
  • [ ] ERP API 블로킹 이슈의 구체적 서술
  • [ ] 의사결정 필요 사항의 명확한 도출
  • [ ] 주간 보고 vs 경영진 보고의 차이
  • [ ] 개발자 병가 대응의 현실적 계획
  • [ ] 보고서의 "5분 내 상황 파악" 가능 여부

PM-04. 회고 (Retrospective) 진행 & 문서화 ⭐⭐ ​

시나리오 배경 ​

프로젝트 완료 후 또는 스프린트 종료 후 회고를 설계하고 결과를 문서화.

프롬프트 ​

프로젝트 완료 후 회고를 설계하고 가상 결과를 문서화해줘.

프로젝트: 3개월간 진행한 "모바일 앱 리뉴얼" 프로젝트
- 결과: 2주 지연 후 출시, 출시 후 크리티컬 버그 1건 핫픽스
- 팀: PM 1, FE 3, BE 2, 디자인 1, QA 1 (총 8명)
- 분위기: 다들 피곤하지만 결과물에는 만족. 그러나 과정에서 불만 있었음

산출물:

1. `pm/retrospective-facilitation-guide.md` — 회고 진행 가이드
   - 목적: 비난이 아닌 학습, 다음에 더 잘하기 위해
   - 시간: 2시간
   - 장소: 오프라인 (편안한 공간)
   - 퍼실리테이터: PM (또는 외부)
   - 그라운드 룰:
     - Vegas Rule (여기서 한 말은 여기서만)
     - 사람이 아닌 프로세스에 초점
     - 모두 발언, 한 사람이 독점 금지

   진행 순서:
   - 체크인 (5분): 이 프로젝트를 한 단어로?
   - 타임라인 (20분): 주요 사건을 시간순으로 포스트잇
   - 잘한 것 (20분): Keep — 계속할 것
   - 개선할 것 (20분): Problem — 문제였던 것
   - 시도할 것 (20분): Try — 다음에 해볼 것
   - 투표 (10분): 가장 중요한 개선안 3개 선정
   - 액션 아이템 (15분): 누가 언제까지 무엇을
   - 체크아웃 (5분): 한 문장으로 마무리

2. `pm/retrospective-result.md` — 가상 회고 결과

   타임라인 (주요 사건):
   - W1: 킥오프, 요구사항 확정 (순조로움)
   - W3: 디자인 시안 3차 수정 (클라이언트 피드백 지연)
   - W5: BE API 설계 변경 (요구사항 추가 발견)
   - W8: 통합 테스트 시작, 버그 폭발 (QA 인력 부족)
   - W10: 출시 1주 전, 야근 시작
   - W12: 2주 지연 출시, 핫픽스 대응

   Keep (잘한 것, 가상):
   - 매일 15분 스탠드업 → 이슈 빠른 공유
   - 디자이너-FE 페어링 → 시안↔구현 갭 최소화
   - Git Flow 전략 → 배포 안정적

   Problem (문제, 가상):
   - QA가 1명이라 통합 테스트 병목
   - 요구사항 변경 관리가 안 됨 (구두 → 구현 → 나중에 문서화)
   - 출시 2주 전부터 야근 → 번아웃, 실수 증가
   - 클라이언트 피드백 루프가 느림 (결재 라인 문제)

   Try (시도할 것, 가상):
   - QA 자동화 도입 (다음 프로젝트)
   - 변경 요청은 Jira 티켓 필수 (구두 금지)
   - "출시 2주 전 코드 프리즈" 정책
   - 클라이언트 측 전담 PO 요청

   액션 아이템:
   | 항목 | 담당 | 기한 | 상태 |
   |------|------|------|------|
   | QA 자동화 도구 벤치마킹 | QA 이수진 | 4/30 | 예정 |
   | 변경관리 프로세스 문서화 | PM 김태호 | 4/15 | 진행중 |
   | 코드 프리즈 정책 초안 | PL 박준영 | 4/20 | 예정 |

3. `pm/lessons-learned-register.md` — 교훈 등록부
   - 이 프로젝트에서 배운 것들을 조직 자산으로
   - 카테고리: 기획, 설계, 개발, 테스트, 관리, 커뮤니케이션
   - 각 교훈: 상황, 교훈, 추천 액션
   - 다음 프로젝트에서 반드시 참고할 것 TOP 3

  • [ ] 그라운드 룰의 현실적 중요성

PM-05. 요구사항 정의서 ⭐⭐ ​

시나리오 배경 ​

고객 요구를 체계적으로 정의하고 합의하는 문서.

프롬프트 ​

아래 서비스의 요구사항 정의서를 작성해줘.

서비스: 사내 지식 관리 시스템 (Internal Wiki)
고객: 우리 회사 내부 (경영지원팀 요청)
사용자: 전 직원 200명

요청사항 (경영지원팀 원문):
"직원들이 쉽게 정보를 찾을 수 있는 위키를 만들어주세요. 
기존에는 메일이나 드라이브에 흩어져 있어서 찾기가 어려워요.
신입사원이 와도 알아서 찾아볼 수 있으면 좋겠고,
부서별로 관리할 수 있어야 하고, 검색이 잘 되어야 해요.
아, 그리고 보안도 중요해요. 아무나 다 보면 안 되는 문서도 있거든요."

산출물: `pm/requirements-definition.md`

1. 요구사항 개요
   - 프로젝트 배경 & 목적
   - 이해관계자 목록
   - 용어 정의

2. 기능 요구사항 (FR)
   - 각 요구사항: ID, 제목, 설명, 우선순위(Must/Should/Could/Won't), 수용 기준
   
   카테고리별:
   A. 문서 관리 (10개 이상)
   - FR-001: 문서 생성 (마크다운 에디터, 이미지/파일 첨부)
   - FR-002: 문서 수정 (버전 관리, 변경 이력)
   - FR-003: 문서 삭제 (soft delete, 관리자만 영구 삭제)
   - FR-004: 문서 트리 구조 (부서 > 카테고리 > 문서)
   - ...
   
   B. 검색 (5개 이상)
   - 전문 검색, 필터 (부서, 날짜, 작성자), 검색어 자동완성
   
   C. 권한 관리 (5개 이상)
   - 부서별 접근 제어, 문서 레벨 권한, 관리자 역할
   
   D. 사용자 경험 (5개 이상)
   - 최근 문서, 즐겨찾기, 읽기 확인, 알림
   
3. 비기능 요구사항 (NFR)
   - 성능: 검색 응답 2초 이내, 동시 접속 100명
   - 보안: SSO 연동, 접근 로그, 데이터 암호화
   - 가용성: 99.5%
   - 접근성: 모바일 웹 지원

4. 제외 범위 (Out of Scope)
   - 명시적으로 이번에 안 하는 것

5. 우선순위 매트릭스
   - MoSCoW 기법으로 전체 정리
   - MVP 범위 = Must + Should의 일부

평가 포인트 ​

  • [ ] 모호한 요청에서 구체적 FR을 도출하는 능력
  • [ ] "검색이 잘 되어야 해요" → 구체적 검색 요구사항 5개
  • [ ] 수용 기준(Acceptance Criteria)의 테스트 가능성
  • [ ] 비기능 요구사항의 포괄성
  • [ ] MoSCoW 분류의 합리성
  • [ ] Out of Scope의 명시적 정의
  • [ ] 전체 ID 체계의 일관성
