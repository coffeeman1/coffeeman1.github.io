CAT-14. 고객 서비스 & 운영 ​

평가 핵심: 에이전트가 CS 응대 스크립트, FAQ, SOP, 클레임 처리 등 고객 접점 업무를 현실감 있게 작성할 수 있는가?

이 카테고리가 중요한 이유 ​

고객 서비스는 공감 + 정확성 + 일관성이 동시에 필요하다. 같은 불만이라도 상황에 따라 대응이 달라져야 하고, 그러면서도 회사 정책의 틀을 벗어나면 안 된다. 에이전트가 이 미묘한 균형을 잡는지 평가한다.

CS-01. CS 응대 스크립트 매뉴얼 ⭐⭐⭐ ​

시나리오 배경 ​

신규 CS 담당자가 바로 업무에 투입될 수 있는 수준의 응대 매뉴얼.

프롬프트 ​

이커머스 고객센터 응대 스크립트 매뉴얼을 만들어줘.

서비스: 온라인 패션 쇼핑몰 "스타일릭"
- 하루 문의: 약 200건
- 채널: 전화 40%, 카카오톡 채팅 35%, 이메일 20%, SNS DM 5%
- 주요 문의: 배송(35%), 교환/반품(30%), 상품문의(20%), 결제(10%), 기타(5%)

매뉴얼:

1. 기본 응대 원칙
   - 인사말 / 종료 인사 (채널별 차이)
   - 전화: "안녕하세요, 스타일릭 고객센터 OOO입니다"
   - 채팅: 좀 더 가볍지만 정중하게
   - 대기 시간 안내 표준 멘트
   - 이관 시 멘트 ("담당 부서로 연결해드리겠습니다")

2. 상황별 스크립트 (각 상황에 3~4개 응대 흐름)

   A. 배송 관련
   - "배송이 언제 오나요?" → 조회 후 안내
   - "배송이 늦어요/안 와요" → 사과 + 확인 + 대안
   - "배송지 변경하고 싶어요" → 가능/불가능 조건별 분기
   - "부분 배송 왔는데 나머지는?" → 분리배송 안내

   B. 교환/반품
   - "사이즈가 안 맞아요" → 교환 절차
   - "상품이 불량이에요" → 사진 요청 → 교환/환불 분기
   - "반품하고 싶은데 7일 지났어요" → 정책 안내 (예외 처리 기준)
   - "교환인데 원래 상품은 언제 수거해가나요?" → 수거 일정

   C. 감정적 고객 대응
   - 화가 많이 난 고객 → 감정 인정 → 사실 확인 → 해결안 제시
   - "다시는 안 사" → 이탈 방지 멘트 (쿠폰 제공 가능 조건)
   - 욕설/폭언 고객 → 경고 → 2차 경고 → 통화 종료 절차
   - 반복 민원 고객 → 이력 확인 → 상급자 이관 기준

   D. 결제/환불
   - "결제가 안 돼요" → PG사별 체크포인트
   - "환불은 언제 되나요?" → 결제수단별 환불 소요 기간
   - "이중결제 됐어요" → 긴급 대응 프로세스

3. 금지 표현 & 대체 표현 20쌍
   - "규정상 안 됩니다" → "고객님 상황을 확인해보겠습니다"
   - "저는 모릅니다" → "담당 부서에 확인해서 안내드리겠습니다"
   - "원래 그렇습니다" → "현재 저희 정책은 ~이며, 고객님 상황에 맞게 확인하겠습니다"

4. 이관/에스컬레이션 기준
   - 담당자가 해결할 수 있는 범위 (금액, 권한)
   - 팀장 이관 기준 (반복 민원, 법적 언급, 금액 50만원 이상)
   - 법무 이관 기준 (소송 언급, 소비자원 신고 언급)

파일: `cs/customer-service-manual.md`

CS-02. FAQ & 헬프센터 콘텐츠 ⭐⭐ ​

시나리오 배경 ​

고객이 스스로 답을 찾을 수 있는 셀프서비스 콘텐츠.

프롬프트 ​

SaaS 서비스의 헬프센터 FAQ 콘텐츠를 만들어줘.

서비스: "팀플로우" — 원격 팀 협업 도구 (화상회의 + 프로젝트 관리 + 채팅)
- 유료 고객: 500사 (주로 10~50인 기업)
- 월 CS 문의 600건 중 40%가 반복 질문

기존 반복 문의 TOP 20 기반으로 FAQ를 작성해.

카테고리 분류:
1. 시작하기 (5문항)
   - 가입, 팀 생성, 멤버 초대, 초기 설정, 앱 설치
   
2. 요금 & 결제 (5문항)
   - 요금제 차이, 결제 방법, 청구서, 환불, 플랜 변경
   
3. 화상회의 (4문항)
   - 회의 생성, 화면 공유, 녹화, 접속 오류
   
4. 프로젝트 관리 (3문항)
   - 보드 생성, 권한 설정, 외부 공유
   
5. 계정 & 보안 (3문항)
   - 비밀번호 재설정, 2FA, 데이터 내보내기

각 FAQ 항목:
- 질문 (고객이 실제로 검색할 법한 자연스러운 문장)
- 답변 (단계별, 스크린샷 위치 명시: "[설정 > 팀 관리 > 멤버 초대]")
- 관련 FAQ 링크 추천 2개
- 이래도 해결 안 되면: "고객센터 문의하기" CTA

추가:
- `cs/troubleshooting-guide.md` — 자주 발생하는 기술 문제 해결 가이드 5개
  - "회의 연결이 안 돼요" → 네트워크 체크 → 브라우저 → 방화벽 → ...
  - 각 가이드: 증상 → 원인 3가지 → 해결 순서도 (YES/NO 분기)

파일:
- `cs/faq-helpcenter.md`
- `cs/troubleshooting-guide.md`

  • [ ] 답변의 단계별 명확성 (1. 클릭 2. 선택 3. 완료)
  • [ ] 화면 경로 표기의 일관성

CS-03. 클레임 처리 & 보상 정책 ⭐⭐⭐ ​

시나리오 배경 ​

고객 불만/클레임에 대한 체계적 처리 프로세스와 보상 기준.

프롬프트 ​

고객 클레임 처리 프로세스와 보상 정책을 만들어줘.

서비스: 음식 배달 플랫폼 "배달잇"
- 일 주문: 5만건
- 일 클레임: 약 500건 (1%)
- 클레임 유형:
  - 배달 지연 (30분 이상): 40%
  - 음식 누락/오배달: 25%
  - 음식 품질 (식은 음식, 쏟아짐): 20%
  - 결제 오류: 10%
  - 위생 문제 (이물질): 5%

산출물:

1. `cs/claim-process.md` — 클레임 처리 프로세스
   - 접수 → 분류 → 조사 → 판단 → 보상 → 종결 → 사후관리
   - 각 단계별 담당자, SLA (응답 시간), 필요 조치
   - 긴급도 분류:
     - Level 1 (즉시): 위생/이물질 → 즉시 환불 + 식품안전 보고
     - Level 2 (4시간): 오배달/누락 → 재배달 or 환불
     - Level 3 (24시간): 지연, 품질 → 쿠폰 or 부분 환불
   - 판단 기준표 (어떤 증거가 있으면 인정, 없으면 거절)

2. `cs/compensation-policy.md` — 보상 기준표
   - 클레임 유형 × 심각도별 보상 매트릭스
   
   | 유형 | 경미 | 보통 | 심각 |
   |------|------|------|------|
   | 배달 지연 (30~60분) | 쿠폰 2,000원 | 쿠폰 3,000원 | - |
   | 배달 지연 (60분+) | - | 배달비 환불 | 전액 환불 |
   | 음식 누락 (1개) | 해당 메뉴 환불 | 해당+쿠폰 | - |
   | 오배달 | - | 전액 환불+쿠폰 | 전액 환불+쿠폰 5,000원 |
   | 품질 (식음) | 쿠폰 2,000원 | 부분 환불 | 전액 환불 |
   | 이물질 | - | - | 전액 환불+쿠폰 10,000원+식약처 보고 |
   
   - 월 보상 한도 (남용 방지): 동일 고객 월 3회 이상 → 검토 대상
   - 음식점 귀책 vs 배달 귀책 구분 기준
   - 보상 비용 분담 (플랫폼 vs 음식점 vs 라이더)

3. `cs/claim-response-templates.md` — 클레임 유형별 응대 템플릿 10개
   - 각 템플릿: 첫 응대, 조사 중 안내, 결과 통보 (인정/거절)
   - 거절 시 부드럽지만 명확한 표현
   - "소비자원에 신고하겠다" 대응 템플릿
   - SNS 공개 불만 대응 (DM으로 전환 유도)

4. `cs/claim-analytics-framework.md` — 클레임 분석 프레임워크
   - 주간/월간 리포트 구조
   - 추적해야 할 지표: CSAT, 해결 시간, 재접수율, 보상 비용
   - 클레임 패턴 분석 (특정 음식점/지역/시간대 집중?)
   - 개선 제안 연결 (데이터 → 운영 개선)

CS-04. SOP (표준운영절차) ⭐⭐ ​

시나리오 배경 ​

반복 업무를 누구나 동일하게 처리할 수 있게 하는 표준 절차서.

프롬프트 ​

아래 3가지 업무의 SOP를 만들어줘. 신입 직원이 혼자 보고 따라할 수 있어야 해.

SOP 1: 신규 고객 온보딩
- 서비스: B2B SaaS (프로젝트 관리 도구)
- 트리거: 유료 결제 완료
- 절차:
  1. 환영 이메일 발송 (24시간 내)
  2. 온보딩 미팅 일정 잡기 (3일 내)
  3. 미팅 진행 (30분, 화상) — 체크리스트 기반
  4. 초기 세팅 지원 (팀 구성, 프로젝트 생성)
  5. 1주 후 체크인 전화
  6. 2주 후 설문 발송
  7. 1개월 후 활용도 리뷰
- 각 단계: 담당자, 사용 도구, 템플릿/스크립트, 체크리스트, 예외 처리

SOP 2: 장애 대응
- 서비스: 위 동일 SaaS
- 트리거: 모니터링 알림 or 고객 신고
- 절차:
  1. 장애 확인 (5분 내)
  2. 심각도 판단 (전체 다운 vs 일부 기능 vs 성능 저하)
  3. 내부 알림 (슬랙 #incident 채널)
  4. 고객 공지 (상태 페이지 업데이트)
  5. 복구 작업 + 진행 상황 업데이트 (30분마다)
  6. 복구 완료 공지
  7. 사후 분석 (Post-mortem) 일정 잡기
- 심각도별 에스컬레이션 기준, 대응 시간 SLA

SOP 3: 계약 갱신 프로세스
- 트리거: 계약 만료 90일 전
- 절차:
  1. D-90: 갱신 대상 목록 추출 + 담당자 배정
  2. D-75: 고객 사용 현황 분석 (활용도, 만족도, 이슈 이력)
  3. D-60: 갱신 제안 미팅 (가격, 업셀 기회)
  4. D-45: 계약서 초안 발송
  5. D-30: 협상 및 조건 확정
  6. D-14: 최종 계약서 서명
  7. D-7: 갱신 완료 확인 + 감사 메일
  8. 미갱신 시: 이탈 분석 + 인터뷰 요청

각 SOP:
- 플로우차트 (텍스트 기반)
- 단계별 상세 절차 (체크리스트 형태)
- 예외 상황 분기 (고객 응답 없음, 일정 조율 불가 등)
- 사용 도구/시스템 명시
- 담당자 역할 (누가 뭘 하는지)
- KPI (각 SOP의 성공 측정 지표)

파일: `cs/sop-collection.md`

CS-06. VOC(고객의 소리) 분석 보고서 ⭐⭐ ​

시나리오 배경 ​

수집된 고객 피드백을 분류·분석하여 인사이트를 도출.

프롬프트 ​

아래 고객의 소리(VOC) 데이터를 분석하고 보고서를 만들어줘.

먼저 `cs/voc-data.json`에 가상 VOC 데이터 100건 생성:
- 수집 기간: 2024년 3월
- 채널: 전화(30), 채팅(25), 이메일(20), 앱 리뷰(15), SNS(10)
- 분류: 칭찬(15%), 건의(25%), 불만(45%), 문의(15%)
- 서비스: 온라인 식료품 배달 "프레시마트"
- 각 VOC: id, date, channel, category, sentiment(positive/neutral/negative), content(한국어 1~3문장), resolved(true/false)

현실적 VOC 예시:
- "새벽배송 항상 정시에 와서 좋아요" (칭찬)
- "유기농 코너 종류 좀 늘려주세요 맨날 품절" (건의)
- "사과를 시켰는데 멍든 게 왔어요 사진 첨부합니다" (불만)
- "적립금 유효기간이 너무 짧아요 3개월은 해주세요" (건의)
- "배달왔는데 냉동식품이 다 녹아있었어요" (불만/심각)

분석 보고서: `cs/voc-analysis-march-2024.md`

1. 월간 요약
   - 총 접수 건수, 채널별 분포, 카테고리별 분포
   - 전월 대비 변화 (가상 데이터)
   - 감성 분석 (긍정/중립/부정 비율)

2. 주요 이슈 TOP 5
   - 각 이슈: 건수, 대표 VOC 원문 인용, 심각도, 현재 대응 상태
   - 예: "냉동식품 해동 배달" — 12건, 심각, 콜드체인 점검 필요

3. 카테고리별 키워드 분석
   - 불만 키워드: "늦음", "상태", "환불", "안 됨"
   - 건의 키워드: "추가", "더", "원하다", "확대"
   - 칭찬 키워드: "빠른", "신선", "좋아", "편리"

4. 채널별 특성
   - 전화: 긴급 불만 집중
   - 앱 리뷰: 칭찬 비율 높지만 별점 영향 큼
   - SNS: 확산 위험 있는 불만

5. 개선 제안 (VOC → 액션 연결)
   - 즉시 조치 (이번 주): 냉동배송 프로세스 점검
   - 단기 (1개월): 적립금 정책 재검토
   - 중기 (분기): 유기농 SKU 확대

6. 경영진 요약 (1페이지)

분석 스크립트: `cs/analyze_voc.py` (JSON → 키워드 추출 → 분류 → 리포트)

CS-07. 서비스 수준 협약 (SLA) 설계 ⭐⭐ ​

시나리오 배경 ​

B2B 고객에게 제공할 SLA 문서 설계.

프롬프트 ​

B2B SaaS 서비스의 SLA 문서를 만들어줘.

서비스: "데이터브릿지" — 기업용 데이터 통합 플랫폼
- 고객: 중견~대기업 (연 계약 5천만원~3억원)
- 현재: SLA 문서가 없어서 장애 때마다 기준이 모호

SLA 문서:

1. `cs/service-level-agreement.md`

   A. 서비스 가용성
   - 목표: 월 99.9% (연간 다운타임 8.77시간 이내)
   - 측정 방법: 5분 간격 모니터링, 외부 모니터링 포함
   - 계획된 유지보수 제외 (사전 48시간 통보)
   - 가용성 미달 시 크레딧:
     - 99.9~99.5%: 월 이용료 10% 크레딧
     - 99.5~99.0%: 20% 크레딧
     - 99.0% 미만: 30% 크레딧
   - 크레딧 청구 절차

   B. 응답 시간 SLA
   - 긴급(P1): 서비스 전면 장애 → 15분 내 첫 응답, 4시간 내 복구
   - 높음(P2): 핵심 기능 장애 → 1시간 내 응답, 8시간 내 복구
   - 보통(P3): 비핵심 기능 → 4시간 내 응답, 48시간 내 복구
   - 낮음(P4): 문의/개선 요청 → 24시간 내 응답

   C. 데이터 관련 SLA
   - 백업: 일 1회 자동 백업, 30일 보관
   - RPO(복구 시점 목표): 1시간
   - RTO(복구 시간 목표): 4시간
   - 데이터 소유권: 고객 귀속

   D. 보고 & 커뮤니케이션
   - 월간 가용성 리포트 제공
   - 장애 발생 시 실시간 상태 페이지 업데이트
   - 주요 장애 후 5영업일 내 Post-mortem 공유

   E. 제외 사항
   - 고객 귀책 (잘못된 API 호출, 비정상 트래픽 등)
   - 천재지변, 클라우드 제공사 장애
   - 베타 기능

   F. Tier별 SLA 차등
   - Standard: 위 기본 SLA
   - Premium: 99.95%, P1 10분 응답, 전담 TAM
   - Enterprise: 99.99%, P1 5분 응답, 24/7 전담 지원

2. `cs/sla-internal-playbook.md` — 내부 대응 매뉴얼
   - SLA 위반 임박 시 에스컬레이션 절차
   - 크레딧 계산 자동화 방법
   - 월간 리포트 작성 가이드

CS-08. 고객 세그먼트별 리텐션 전략 ⭐⭐⭐ ​

시나리오 배경 ​

고객 데이터를 기반으로 세그먼트별 맞춤 리텐션 전략 수립.

프롬프트 ​

고객 데이터를 분석하고 세그먼트별 리텐션 전략을 만들어줘.

서비스: 구독형 온라인 교육 플랫폼 "런잇"
- MAU: 25,000명
- 유료 구독: 8,000명 (월 19,900원)
- 월 이탈률: 5.5% (업계 평균 4%)
- 평균 구독 기간: 4.2개월 (목표 6개월)

고객 데이터 (가상): `cs/customer-segments.json` 생성
- 8,000명의 유료 고객 데이터
- 필드: id, signup_date, last_login, monthly_logins, courses_completed, 
  courses_in_progress, total_study_hours, support_tickets, nps_score, 
  plan_type(monthly/annual), age_group, referral_source

세그먼트 분석: `cs/retention-strategy.md`

1. 세그먼트 정의 (RFM + 행동 기반)
   - Champion (충성): 매주 접속, 코스 완료 3+, NPS 높음 → 약 15%
   - Engaged (활발): 주 2~3회 접속, 학습 중 → 약 25%
   - At Risk (위험): 접속 빈도 하락, 미완료 코스 방치 → 약 20%
   - Dormant (휴면): 30일+ 미접속, 결제만 유지 → 약 25%
   - New (신규): 가입 30일 이내 → 약 15%

2. 세그먼트별 이탈 위험 분석
   - 각 세그먼트의 이탈률
   - 이탈 직전 행동 패턴 (접속 빈도 변화, 학습 시간 감소)
   - 이탈 사유 추정

3. 세그먼트별 리텐션 전략

   Champion:
   - 전략: 보상 + 커뮤니티 (놓치면 안 되는 고객)
   - 액션: 전용 할인, 얼리 액세스, 커뮤니티 리더 초대, 후기 요청

   Engaged:
   - 전략: 습관 강화 (Champion으로 승격)
   - 액션: 학습 리마인더, 코스 추천, 마일스톤 축하

   At Risk:
   - 전략: 재참여 유도 (가장 중요한 세그먼트!)
   - 액션: "돌아와요" 이메일, 미완료 코스 리마인더, 1:1 상담 제안

   Dormant:
   - 전략: 재활성화 or graceful exit
   - 액션: 할인 쿠폰, 새 콘텐츠 알림, 구독 일시정지 옵션

   New:
   - 전략: 온보딩 강화 (첫 경험이 모든 것을 결정)
   - 액션: 가이드 투어, 첫 코스 추천, 3일/7일/14일 체크인

4. 각 전략별 실행 계획
   - 이메일/푸시/인앱 메시지 타이밍
   - A/B 테스트 계획
   - 성과 측정 KPI (세그먼트별 이탈률 변화)

5. 이탈 예측 모델 설계 (개념)
   - 어떤 변수가 이탈을 예측하는지
   - 조기 경보 기준 (이 지표가 X 이하면 At Risk로 전환)

분석 스크립트: `cs/segment_analysis.py`

평가 포인트 ​

  • [ ] 세그먼트 정의의 현실적 비율
  • [ ] At Risk 세그먼트의 중요성 인식 (이탈 직전 개입)
  • [ ] 세그먼트별 전략의 차별화 (같은 전략 반복 아닌)
  • [ ] 이메일/푸시 타이밍의 구체성
  • [ ] 구독 일시정지라는 graceful exit 전략
  • [ ] 이탈 예측 변수의 합리성
  • [ ] 분석 스크립트의 실행 가능성
