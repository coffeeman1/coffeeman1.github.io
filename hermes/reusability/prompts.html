프롬프트 관리 ​

Reusability CoreContext FilesPrompt Caching

스킬이 "방법(how)"이라면 프롬프트는 "성향·규칙·맥락(what matters, how to speak)"입니다. Hermes Agent는 프롬프트를 레이어드 파일 시스템으로 관리하여, 동일한 에이전트를 프로젝트·팀·개인별로 다르게 행동하도록 조형합니다.

프롬프트의 3계층 구조 ​

Hermes Agent는 매 턴마다 시스템 프롬프트를 동적으로 재조립합니다. 조립에 들어가는 레이어는 다음과 같습니다.

┌───────────────────────────────────────────────────┐
│  Layer 4 ─ 실행 컨텍스트                            │
│  활성 스킬 본문 · 도구 카탈로그 · 기억 스냅샷       │
├───────────────────────────────────────────────────┤
│  Layer 3 ─ 프로젝트 레벨 (AGENTS.md, context.yaml)  │
│  이 레포의 컨벤션·아키텍처·금기사항                 │
├───────────────────────────────────────────────────┤
│  Layer 2 ─ 사용자 레벨 (~/.hermes/USER.md)          │
│  내 선호도·전문성·언어·응답 스타일                  │
├───────────────────────────────────────────────────┤
│  Layer 1 ─ 페르소나 (~/.hermes/SOUL.md)             │
│  에이전트의 성격·톤·기본 원칙                       │
├───────────────────────────────────────────────────┤
│  Layer 0 ─ 시스템 베이스 (프레임워크 내장)           │
│  도구 호출 규약·오류 처리·안전 정책                 │
└───────────────────────────────────────────────────┘

이 레이어링이 재사용성의 핵심입니다. Layer 0–2는 모든 프로젝트에서 재사용되고, Layer 3은 프로젝트와 함께 git에 커밋되어 팀원 간 공유됩니다.

"한 번 작성한 프롬프트를 여러 상황에서 재사용한다"는 것은, 프롬프트를 **파일로 쪼개고 계층화하는 것**에서 시작합니다. 긴 단일 프롬프트는 재사용할 수 없습니다.

Layer 1 · SOUL.md — 에이전트 페르소나 ​

SOUL.md는 에이전트의 일관된 성격을 정의합니다. 여러 프로젝트에서 같은 SOUL.md를 씁니다.

markdown
# Hermes · 기본 페르소나

## 정체성
나는 Hermes, 소프트웨어 엔지니어링 파트너다.
신중하고, 근거 기반으로 판단하며, 확실하지 않은 것은 확실하지 않다고 말한다.

## 커뮤니케이션 원칙
- 답을 먼저, 설명은 필요할 때만.
- 코드는 실행 가능하게, 추측은 표시해서.
- 무엇을 했는지보다 **왜** 그렇게 했는지 남긴다.

## 작업 원칙
- 가역적이지 않은 작업은 반드시 사전 확인한다.
- 테스트가 없으면 작업을 마쳤다고 말하지 않는다.
- 엣지 케이스는 생각하기 전에 먼저 찾는다 (grep, 기존 테스트 확인).

## 금기사항 (do-not)
- 요청 범위를 넘는 리팩터링을 하지 않는다.
- 사용자 코드에 이모지를 넣지 않는다.
- "완료했습니다" 끝맺음으로 요약을 대체하지 않는다.

SOUL.md는 짧게

길면 길수록 매 턴 토큰을 먹습니다. 300–500토큰 이내로 유지하고, 세부 규칙은 USER.md나 AGENTS.md로 내리세요.

Layer 2 · USER.md — 사용자 프로필 ​

USER.md는 Hermes가 당신에 대해 학습한 것입니다. 메모리 넛지가 자동 업데이트하지만, 직접 편집해도 됩니다.

markdown
# User Profile

## 전문성
- Python/TypeScript 8년, Go 3년
- AWS 인프라, k8s 운영 경험
- ML 파이프라인보다는 데이터 API 쪽

## 선호 스타일
- 한국어 응답 (코드 주석은 영어 OK)
- 불필요한 서론 생략
- 제안은 항상 트레이드오프 포함

## 반복 컨텍스트
- 주 IDE는 VS Code, 쉘은 zsh
- 로컬은 macOS, 배포는 Linux
- 주로 오전에 코딩, 오후는 리뷰

USER.md는 모든 프로젝트에서 로드되므로 특정 레포에 묶이는 규칙은 넣지 말고 AGENTS.md로 옮기세요.

Layer 3 · AGENTS.md — 프로젝트 컨텍스트 (가장 중요) ​

프로젝트 디렉토리에 .hermes/AGENTS.md를 두면 에이전트가 그 레포에서 작업할 때만 자동 주입합니다. 팀과 함께 쓰는 가장 강력한 재사용 단위입니다.

markdown
# my-web-service · Agent Context

## 프로젝트 개요
React + Next.js 14 + Prisma + PostgreSQL.
E2E는 Playwright, 유닛은 Vitest.

## 아키텍처 규약
- 서버 컴포넌트가 기본, 클라이언트 컴포넌트는 필요시에만
- DB 접근은 `src/server/db.ts`를 거쳐야 함 (직접 prisma 호출 금지)
- API 라우트는 `app/api/[...]/route.ts`로만 추가

## 변경 원칙
- 마이그레이션은 `prisma migrate dev`로 생성, 수동 편집 금지
- main 브랜치 직접 푸시 금지, PR 필수
- 스키마 변경은 seed 데이터도 같이 업데이트

## 자주 틀리는 것 (에이전트 주의)
- `use client` 디렉티브 누락
- Prisma include 깊이 3단계 이상 — N+1 주의
- 환경변수는 `env.mjs`를 거쳐서 사용 (process.env 직접 금지)

## 테스트 명령
- 유닛: `pnpm test:unit`
- E2E: `pnpm test:e2e`
- 타입 체크: `pnpm typecheck`
- 빌드: `pnpm build`

## 배포
- Preview: PR 머지 시 Vercel 자동
- Production: `main`에 머지 후 `pnpm deploy:prod`

AGENTS.md 설계 체크리스트 ​

반드시 포함

  • 아키텍처 경계·모듈 책임
  • 코드에 드러나지 않는 규약
  • 자주 틀리는 패턴(grep하면 나오지 않는 암묵지)
  • 테스트/빌드/배포 명령
  • 파괴적 작업의 안전장치

넣지 말 것

  • README에 이미 있는 소개 문구
  • API 키·시크릿·내부 URL
  • 쉽게 변하는 버전 번호 (의존성 목록)
  • grep 두 번이면 알 수 있는 세부 파일 경로
  • "친절하게 답해주세요" 같은 스타일 지시 (SOUL.md로)

커밋하세요

.hermes/AGENTS.md는 레포에 커밋하는 것을 전제로 설계되었습니다. 팀원이 각자의 에이전트를 쓸 때 동일한 규약이 자동 적용됩니다. 이것이 "레포 자체가 에이전트 온보딩 문서"가 되는 핵심입니다.

context.yaml — 프로젝트 설정 오버라이드 ​

AGENTS.md 옆에 context.yaml을 두면 프로젝트 단위로 설정을 오버라이드할 수 있습니다.

yaml
# .hermes/context.yaml
model: openrouter/anthropic/claude-3.5-sonnet    # 이 프로젝트는 Claude로

tools:
  enabled: [code-execution, file-management, shell-execute]
  disabled: [image-generation, tts]              # 불필요 도구 차단

security:
  approval_required: true
  allowlist:
    commands:
      - "pnpm test:*"
      - "pnpm typecheck"
      - "prisma migrate dev*"

skills:
  prefer: [release-flow, code-review, github-pr-review]
  disable: [web-scraping]

memory:
  scope: per-directory                          # 이 레포 전용 세션

프롬프트 조각 재사용 (prompts/ 디렉토리) ​

긴 AGENTS.md가 중복되기 시작하면 조각으로 쪼개고 참조하세요.

.hermes/
├── AGENTS.md
├── context.yaml
└── prompts/
    ├── conventions.md       # 공통 코드 규약
    ├── testing_rules.md     # 테스트 작성 규칙
    ├── review_checklist.md  # 리뷰 체크리스트
    └── incident_playbook.md # 장애 대응 플레이북

AGENTS.md에서 include 문법으로 참조:

markdown
## 코드 규약
{{> prompts/conventions.md }}

## 테스트
{{> prompts/testing_rules.md }}

스킬 체이닝과 동일한 {{> path }} 문법입니다 — 스킬과 프롬프트가 같은 조립 규칙을 씁니다.

슬래시 명령어로 즉시 재사용 ​

자주 쓰는 프롬프트는 스킬화할 필요 없이 슬래시 명령으로 노출할 수 있습니다. ~/.hermes/commands/ 혹은 .hermes/commands/에 .md 파일을 두면 즉시 /파일이름으로 호출됩니다.

markdown
<!-- .hermes/commands/review.md -->
---
description: 현재 diff를 이 레포 규약 기준으로 리뷰한다
---

`git diff --staged`의 변경사항을 리뷰하라. 체크리스트:

1. **테스트** — 신규 로직에 테스트가 있는가? 실패 경로 포함?
2. **규약** — AGENTS.md의 자주 틀리는 것 목록과 대조
3. **성능** — N+1, O(n²), 불필요한 await
4. **보안** — env 직접 접근, 사용자 입력 신뢰, SQL concat
5. **이름** — 변수/함수 이름이 의도를 드러내는가

각 항목에 ✅/⚠️/❌와 파일:줄 번호. 마지막에 우선순위 3가지.

사용:

bash
> /review

슬래시 명령 vs 스킬

슬래시 명령은 프롬프트 템플릿에 가깝습니다 — 단순히 한 번의 복잡한 지시를 저장. 스킬은 도구 호출 워크플로우 + 매개변수 + 진화 이력. 작업이 결정론적이고 재현 가능해지는 순간이 슬래시 명령에서 스킬로 승격할 타이밍입니다.

자세한 것은 슬래시 명령어 & 컨텍스트 참조.

프롬프트 캐싱 (Prompt Caching) ​

Hermes는 프롬프트 캐싱을 지원하는 프로바이더(Anthropic, OpenAI 일부)에서 자동으로 캐시 히트를 활용합니다. 캐싱 효과를 극대화하려면:

안정된 레이어 먼저
SOUL.md → USER.md → AGENTS.md 순으로 주입 (변하지 않는 것부터)
동적 부분은 끝에
현재 파일 내용, 도구 결과는 마지막에 — 앞부분이 캐시 적중
레이어 경계 고정
SOUL.md를 매 턴 수정하면 캐시가 매번 무효화
TTL 주의
Anthropic 캐시는 5분 TTL — 긴 사고 시간 사이에 끊기면 재구축 비용
yaml
# ~/.hermes/config.yaml
caching:
  enabled: true
  breakpoints:
    - after: soul           # SOUL.md 뒤에 캐시 경계
    - after: user           # USER.md 뒤에 캐시 경계
    - after: agents         # AGENTS.md 뒤에 캐시 경계
  log_hits: true             # /usage 출력에 캐시 적중률 표시

/usage에서 캐시 적중률을 확인할 수 있습니다.

📊 세션 사용량:
  입력 토큰:       12,450
  캐시 적중:       9,120 (73%)
  캐시 쓰기:       3,330
  출력 토큰:       2,800
  예상 비용:       $0.014  (캐싱으로 68% 절감)

프롬프트 버전 관리 ​

AGENTS.md·SOUL.md는 코드처럼 버전 관리하세요.

bash
# 프로젝트 레벨
git log --follow .hermes/AGENTS.md        # 이 레포의 컨벤션 변화사
git blame .hermes/AGENTS.md               # "이 규칙 누가 넣었나"

# 사용자 레벨
cd ~/.hermes && git init                  # dotfiles처럼 개인 버전 관리
git add SOUL.md USER.md commands/
git commit -m "tighten review checklist"

에이전트는 프롬프트가 바뀐 시점을 인지하지 못하므로, 규약을 크게 바꿨다면 대화 시작 시 /reset으로 새 세션을 여는 것이 좋습니다.

환경별 프롬프트 분기 ​

동일 레포라도 로컬·CI·스테이징에서 다른 제약이 필요할 수 있습니다.

yaml
# .hermes/context.yaml
profiles:
  local:
    security: { approval_required: false }
    prompts: [prompts/conventions.md, prompts/dev_tips.md]
  ci:
    security: { approval_required: true, allowlist: { commands: ["pnpm test:*"] } }
    prompts: [prompts/conventions.md, prompts/ci_strict.md]
    tools: { disabled: [shell-execute, file-write] }
  production:
    security: { approval_required: true, dm_pairing: true }
    prompts: [prompts/incident_playbook.md]

프로파일 활성화:

bash
hermes --profile ci
# 또는
export HERMES_PROFILE=production

프롬프트 품질 체크리스트 ​

작성·리뷰 전에 다음을 확인하세요.

  • [ ] 재사용 가능한가? 한 번만 쓸 거면 그냥 대화로 입력
