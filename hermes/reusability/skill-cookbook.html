목차

스킬 쿡북 · 제작부터 적용까지 ​

PracticalStep-by-StepKorean

스킬 저작 가이드가 "처음 만드는 법" 튜토리얼이라면, 이 쿡북은 **"실전에서 어떻게 설계하고 어디에 적용하는가"**에 집중합니다. 문서 작성 계열 스킬을 예시로 들되 원칙은 모든 도메인에 적용됩니다. 바로 가져다 쓸 수 있는 완성된 스킬 12종은 스킬 예시집에 있습니다.

이 문서를 읽고 나면 ​

  • 스킬로 자동화할 작업을 올바르게 고를 수 있다
  • 스킬 번들의 5가지 유형을 구분하고 각각에 맞는 설계를 할 수 있다
  • 작성 → 테스트 → 검증 → 배포의 전체 파이프라인을 돌릴 수 있다
  • 만든 스킬을 CLI, 크론, 게이트웨이, API 서버, Web UI, 서브에이전트 6가지 경로에서 호출할 수 있다
  • 팀·조직 단위로 공유·유지보수하는 전략을 세울 수 있다

Part 1 · 만들기 전에 결정할 것 ​

스킬을 작성하기 전에 꼭 정리해야 할 네 가지 질문입니다.

질문 1 — 정말 스킬이 필요한가 ​

스킬이 정답

  • 같은 절차를 5번 이상 반복했다
  • 여러 도구를 일정한 순서로 호출한다
  • 결과 형식이 예측 가능하다
  • 팀원과 공유할 가치가 있다
  • 매개변수 2–4개로 일반화 가능하다

다른 선택이 나음

  • 한 번만 쓸 작업 → 일반 대화
  • 매개변수 없는 지시 반복 → 슬래시 명령
  • 성격·규약 정의 → SOUL.md / AGENTS.md
  • 외부 시스템 API → MCP 서버
  • 동작 가로채기·감사 → 플러그인

질문 2 — 어느 범위(scope)에 둘 것인가 ​

범위경로공유적합한 경우
개인 전역~/.hermes/skills/<name>/개인만실험·개인 워크플로우
프로젝트<proj>/.hermes/skills/<name>/팀(git)레포별 규약 의존
프로파일~/.hermes/profiles/<p>/skills/<name>/프로파일별 격리클라이언트·환경 분리
공개agentskills.io 허브전체 커뮤니티범용 가치 있는 스킬

범위를 바꾸는 순서

대부분의 좋은 스킬은 개인 → 프로젝트 → 공개로 진화합니다. 처음부터 공개용으로 만들지 말고, 본인이 먼저 2–3주 써보며 다듬으세요.

질문 3 — 어떤 유형인가 (5가지 archetype) ​

스킬은 대개 다음 다섯 유형 중 하나에 속합니다. 유형을 먼저 고르면 설계가 쉬워집니다.

Type A
📝

변환 (Transformer)

입력 자료를 다른 형식으로 바꿈. 대부분의 "문서 생성" 스킬이 여기. 커밋→체인지로그, 회의 녹취→회의록, 코드→API 문서.

예: changelog-generator결정론적
Type B
🔍

수집 + 요약 (Collector)

여러 출처에서 정보를 모아 정리. 뉴스레터, 리서치 브리핑, 이슈 트리아지.

예: oncall-triageweb + file
Type C
🚀

실행 (Executor)

실제 부작용 있는 작업. 배포, 마이그레이션, 환경 구성. 가장 위험하므로 승인·테스트 엄격.

예: deploy-to-ecr부작용 有
Type D
✅

검증 (Validator)

기존 결과물을 검사하여 리포트 생성. 코드 리뷰, 보안 감사, 문서 일관성 검증.

예: docs-consistency-checkread-only
Type E
🧩

합성 (Composer)

다른 스킬들을 엮는 상위 스킬. {{> sub-skill}} 문법으로 하위 호출.

예: release-flow오케스트레이션
Bonus
🧠

학습형 (Learned)

자주 반복되는 대화를 학습 루프가 자동 추출한 결과. 사용자가 리파인해서 정식 스킬로 승격.

질문 4 — 입력·출력 계약은 명확한가 ​

설계 단계에서 종이에 먼저 써보세요. 본문을 쓰기 시작하면 이미 늦습니다.

스킬 이름: pr-description-writer
유형:     Type A (변환)

입력:
  - base_ref: string (default "main")
  - head_ref: string (default "HEAD")
  - tone: enum [concise, detailed, casual] (default concise)
  - template: path (optional)

출력:
  - 포맷: Markdown
  - 길이: 200–600단어
  - 구조: ## Summary / ## Changes / ## Test plan / ## Screenshots
  - 파일 경로: /tmp/pr-description-<timestamp>.md
  - 클립보드 복사: 옵션

부작용:    [local_write] (reversible: true)
필요 도구: git-diff, git-log, file-write
승인:     auto

4가지가 명확하지 않으면 아직 스킬로 만들 준비가 안 된 것입니다. 질문을 다듬거나, 몇 번 더 수동으로 돌려보고 패턴이 고정될 때 돌아오세요.

  • 첫 줄: 한 문장으로 "무엇을 하는가"
  • 둘째 줄: 입력·출력의 핵심
  • 셋째 줄(있다면): 경계 또는 주의
  • 한국어·영어 키워드를 같이 넣으면 라우팅 정확도 향상

3단계 · 본문 작성 ​

frontmatter 아래는 에이전트에게 주는 지시문입니다. 사람이 읽는 설명이 아닙니다.

markdown
# PR Description Writer

## 목표
`{{base_ref}}` → `{{head_ref}}` 사이의 변경사항을 사용자 관점 PR 설명으로
변환하고, `{{tone}}` 톤으로 `{{output_path}}`에 저장한다.

## 절차

### 1. 변경 범위 수집
- `git log {{base_ref}}..{{head_ref}} --format="%H%x00%s%x00%b%x00%an"` 실행
- 커밋 수 확인: 0이면 "변경 없음" 리턴
- `git diff --stat {{base_ref}}..{{head_ref}}` 로 파일별 증감 행 수

### 2. 변경 분류
Conventional Commits(type(scope): subject)가 있으면 type별로 그룹:
- feat → "새 기능"
- fix  → "버그 수정"  
- perf → "성능 개선"
- refactor → "리팩터"
- docs → "문서"
- chore → 설명에서 제외

규약에 안 맞으면 파일 경로 패턴으로 추론:
- `tests/` → 테스트 추가
- `docs/`  → 문서
- `*.config.*` → 설정

### 3. Breaking Change 감지
- `BREAKING CHANGE:` 푸터 or `!` 마크 있는 커밋
- 공개 API 파일(`src/api/*`, `*/public/*`) 대규모 변경
- 있으면 별도 "⚠ Breaking Changes" 섹션 상단에

### 4. 문서 구성
템플릿 없을 때 기본 구조:

    # <type 요약, 한 문장>
    
    ## Summary
    <2–4문장으로 이 PR이 해결하는 문제와 접근>
    
    ## Changes
    - <분류된 변경 불릿, 원본 문구 최대 보존>
    
    ## Test plan
    - [ ] <변경 영역별 테스트 체크리스트>
    
    ## Notes (있을 때만)
    <알려진 제한, 후속 작업, 의존성>

`{{template}}`이 있으면 템플릿의 섹션 이름을 따름.

### 5. 톤 적용
- `concise`: 불릿 위주, 최소 200단어
- `detailed`: 각 섹션에 맥락 추가, 400–600단어
- `casual`: 친근한 1인칭 복수 ("we", "우리"), 이모지 사용

### 6. 저장 & 보고
- `{{output_path}}`에 저장
- 워드 카운트, 섹션 수, 포함된 Breaking change 수를 요약 보고
- 클립보드 복사 옵션이 활성화되어 있으면 `pbcopy` / `xclip` / `clip.exe`로

## 의사결정 분기
- **커밋이 50개 넘으면** — scope별로 그룹핑, 세부 내용 축약
- **diff가 500라인 넘으면** — `references/large-diff-strategy.md` 로드
- **테스트 파일 변경 없음** — Test plan에 "기존 테스트로 커버" 언급, 경고
- **BREAKING CHANGE** — 상단에 ⚠ 섹션 필수

## 실패 기준 (do-not)
- 커밋 메시지에 없는 내용을 **날조**하지 말 것
- 변경 파일 경로를 **전체 경로로** 노출하지 말 것 (상대 경로만)
- 민감 정보(.env, keys/)가 diff에 있으면 **경고**하고 중단
- base/head ref가 존재하지 않으면 에러 리턴
- 빈 diff인데 설명을 지어내지 말 것

4단계 · 스크립트 추가 (필요 시) ​

복잡한 파싱 로직은 scripts/로 분리합니다. Python 기준:

python
# scripts/classify_commits.py
"""
커밋 메시지 배열을 받아 Conventional Commits로 분류한다.

Usage (JSON via stdin):
  { "commits": [{"hash":"...", "subject":"feat(api): add oauth"}] }

Output (JSON via stdout):
  {
    "grouped": {"feat": [...], "fix": [...], "docs": [...]},
    "breaking": [...],
    "unclassified": [...]
  }
"""
import json, re, sys

TYPES = ("feat", "fix", "perf", "refactor", "docs", "test", "chore", "style", "build", "ci")
HEADER = re.compile(
    r'^(?P<type>' + '|'.join(TYPES) + r')(?P<scope>\([^)]+\))?(?P<bang>!)?:\s*(?P<subject>.+)$'
)


def classify(commits):
    grouped = {t: [] for t in TYPES}
    breaking = []
    unclassified = []
    
    for c in commits:
        m = HEADER.match(c["subject"].strip())
        if not m:
            unclassified.append(c)
            continue
        
        t = m.group("type")
        entry = {
            "hash": c["hash"],
            "scope": m.group("scope")[1:-1] if m.group("scope") else None,
            "subject": m.group("subject"),
        }
        grouped[t].append(entry)
        
        if m.group("bang") or "BREAKING CHANGE" in c.get("body", ""):
            breaking.append(entry)
    
    grouped = {k: v for k, v in grouped.items() if v}
    return {
        "grouped": grouped,
        "breaking": breaking,
        "unclassified": unclassified,
    }


if __name__ == "__main__":
    data = json.loads(sys.stdin.read())
    result = classify(data["commits"])
    print(json.dumps(result, ensure_ascii=False, indent=2))

frontmatter에 스크립트 시그니처 선언:

yaml
scripts:
  - name: classify-commits
    path: scripts/classify_commits.py
    description: "Conventional Commits 배열을 type별로 분류"
    parameters:
      - { name: commits, type: array, required: true }

# 큰 diff 대응 전략

500라인 이상의 diff를 다룰 때 적용한다.

## 원칙
1. **파일 단위가 아니라 모듈 단위**로 요약 — 개별 파일 나열 금지
2. **신호 대 잡음** — 생성된 파일(.lock, migrations/*.sql)은 축약
3. **계층화** — Summary는 고수준, Changes는 모듈별, Notes에 디테일

## 축약 기준
- lock 파일 변경 → "패키지 업데이트 (N개 패키지)"
- 마이그레이션 → "DB 스키마 변경: <핵심 테이블>"
- 자동 생성 파일(*.generated.*) → 목록에서 제외
- 공백·포맷팅만 있는 변경 → "코드 정리"로 통합

6단계 · 드라이런 ​

bash
hermes skills run pr-description-writer \
  --dry-run \
  base_ref=main \
  head_ref=feature/oauth \
  tone=concise

출력 예:

🔍 Dry run: pr-description-writer@0.1.0

계획된 도구 호출:
  1. shell-execute("git log main..feature/oauth --format=...")
  2. shell-execute("git diff --stat main..feature/oauth")
  3. [스크립트 호출] classify-commits (stdin 파이프)
  4. file-write("/tmp/pr-description.md", <rendered>)

부작용: local_write
승인: auto
Progressive disclosure: SKILL.md만 로드 (조건 없음)

실제 실행하지 않음. --dry-run 제거하여 실행.

7단계 · 테스트 작성 ​

yaml
# tests/test_concise_happy_path.yaml
name: concise tone with conventional commits
given:
  params:
    base_ref: main
    head_ref: HEAD
    tone: concise
    output_path: ./tmp/pr.md
  mocks:
    - cmd: "git log main..HEAD *"
      exit: 0
      stdout: |
        abc123\x00feat(auth): add Google OAuth\x00Adds Google OAuth flow\x00Alice
        def456\x00fix(session): handle null user\x00Fixes #42\x00Bob
    - cmd: "git diff --stat main..HEAD"
      exit: 0
      stdout: "5 files changed, 120 insertions(+), 15 deletions(-)"
expect:
  exit: 0
  files_written:
    - path: ./tmp/pr.md
      contains:
        - "## Summary"
        - "OAuth"
        - "## Changes"
        - "## Test plan"
      not_contains:
        - "chore"           # chore는 concise에서 숨김
        - "날조"             # 없는 내용 지어내지 않음
  summary_contains: ["concise", "2 commits"]

Part 3 · 설계 원칙 10계명 ​

실무에서 반복되는 실수와 해결책.

1. 이름은 동사 + 명사 ​

좋은:  release-notes-gen, pr-description-writer, docs-consistency-check
나쁜:  docs-helper (너무 모호), super-util (유형 불명), my-skill (범용 금지)

2. 매개변수는 5개 이하 ​

5개를 넘으면 다른 스킬로 분리하거나 서브 커맨드로 나누세요. 많은 매개변수는 인터페이스 설계 실패의 신호입니다.

3. 출력 형식을 고정 ​

에이전트가 여러 번 돌려도 같은 구조의 결과가 나오도록 본문에 명시. 이것이 다른 스킬이 체이닝해서 쓸 수 있는 전제 조건입니다.

4. 실패 경로를 문서화 ​

성공 경로만 쓰지 말고 **"이런 경우엔 이렇게"**를 반드시. 실제 학습 가치는 대부분 여기서 나옵니다.

5. 부작용을 태깅 ​

safety.side_effects에 [network, local_write, remote_write, credentials, process_spawn] 중 해당하는 것을 정확히 기재. 이 태깅이 병렬 실행 가능성과 승인 정책을 결정합니다.

6. 가역성 명시 ​

reversible: true면 실패 시 되돌릴 수 있음. 파일 덮어쓰기 전 .bak 만들거나, DB 작업 전 트랜잭션 스냅샷 등.

7. Progressive Disclosure 활용 ​

본문이 2KB 넘어가면 references/로 분리. 베이스 주입량을 작게 유지해야 스킬 수가 늘어도 컨텍스트가 터지지 않습니다.

8. 시크릿 하드코딩 금지 ​

환경변수나 requires.env_vars로 선언. 스킬을 공유할 때 시크릿이 딸려가면 즉시 유출입니다.

9. Tests/가 없으면 Publish 금지 ​

회귀 테스트 없는 스킬은 업데이트할 때마다 다른 사용자를 지뢰밭으로 보냅니다. 최소 happy path + edge case 1개.

10. SemVer 엄격 준수 ​

  • 파괴적 변경 → 메이저 (1.0 → 2.0)
  • 매개변수 추가 → 마이너 (1.0 → 1.1)
  • 본문 개선 → 패치 (1.0.0 → 1.0.1)

Part 4 · 6가지 적용 경로 ​

만든 스킬을 어디서 어떻게 호출할지.

경로 1 · CLI 대화 ​

가장 기본. 자연어로 부르거나 슬래시로 직접.

bash
# 자연어 (라우터가 선택)
> "이 브랜치의 PR 설명을 작성해줘"

# 슬래시로 직접
> /pr-description-writer base_ref=main tone=detailed

# 인자를 자연어와 섞어
> "PR 설명 만들어줘. 톤은 casual로."

CLI는 탐색·개인 사용에 최적. 첫 2–3주는 여기서만 돌려 보세요.

경로 2 · 슬래시 직접 호출 (비대화형) ​

bash
hermes skills run pr-description-writer \
  base_ref=main \
  head_ref=HEAD \
  tone=concise \
  output_path=./PR_DESC.md

스크립트·CI·자동화에 적합. 결과는 파일로.

경로 3 · 크론 (헤드리스) ​

자세한 것은 크론 스케줄러.

경로 5 · API 서버 (프로그래매틱) ​

웹 앱·CI 파이프라인에서 HTTP로 호출. API 서버 참조.

python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8080/v1",
    api_key="hermes-local",
)

resp = client.chat.completions.create(
    model="default",
    messages=[{
        "role": "user",
        "content": "/pr-description-writer base_ref=main head_ref=feature/oauth tone=concise"
    }],
    extra_headers={"X-Hermes-Session-Id": "ci-job-12345"},
)

description = extract_file_content(resp, "/tmp/pr-description.md")

GitHub Actions 예시:

yaml
- name: Generate PR description

Part 5 · 팀 공유 전략 ​

개인 스킬을 팀으로 확장하는 실무 절차.

단계 1 · 레포 이동 ​

bash
# 개인 스킬을 프로젝트 레포로
mv ~/.hermes/skills/pr-description-writer ./.hermes/skills/

# .gitignore 확인 — .hermes/는 커밋해야 함 (팀 공유 목적)

git add .hermes/skills/pr-description-writer
git commit -m "chore: add team skill pr-description-writer"

단계 2 · AGENTS.md에 언급 ​

팀원이 이 스킬의 존재를 발견하게 만들기:

markdown
<!-- .hermes/AGENTS.md 중 -->

## 팀 공통 스킬
이 레포에는 다음 스킬이 포함되어 있습니다. 자동 라우팅되거나 `/스킬이름`으로 호출하세요.

- `/pr-description-writer` — PR 설명 자동 작성
- `/release-notes-gen` — 릴리스 노트 생성
- `/docs-consistency-check` — 문서 일관성 검증

새 스킬 추가는 `.hermes/skills/`에 PR. 리뷰는 #dev-tools 채널에서.

단계 3 · PR 리뷰 체크리스트 ​

팀 스킬에 대한 리뷰 기준을 정해 두세요:

  • [ ] description이 라우팅 가능할 정도로 구체적인가
  • [ ] parameters에 검증·기본값이 있는가
  • [ ] 본문에 do/do-not이 있는가
  • [ ] tests/에 해피 + 엣지 최소 2개
  • [ ] examples/에 1개 이상
  • [ ] 시크릿·내부 URL 없음
  • [ ] 유사 스킬과 중복되지 않음

리파인먼트 반영 ​

학습 루프가 자동으로 제안하는 개선 사항:

사용자: "방금 그 PR 설명에서 테스트 체크리스트가 빠졌어"

에이전트: [실패 패턴 감지 → refinement 초안 생성]
  "pr-description-writer에 개선 사항을 제안드립니다:
   
   - 'tests/' 디렉토리 변경이 없는 PR에서도 Test plan 섹션 필수 생성
   - '기존 테스트로 커버' 문구를 기본값으로 삽입
   
   refinements에 추가하고 v0.1.0 → v0.1.1로 업데이트할까요?"

Deprecate 기준 ​

다음 중 하나라도 해당하면 폐기 검토:

  • 최근 30일 사용량 < 3
  • 성공률 < 0.6
  • 6개월간 수정 없음 + 의존 도구 변경
  • 상위 스킬로 흡수됨
bash
hermes skills deprecate pr-description-writer \
  --reason "replaced by pr-toolkit@2.0" \
  --migrate-to pr-toolkit

Part 7 · 자주 마주치는 함정 ​

안티패턴

  • 메가 스킬 — 여러 관심사를 한 스킬에 몰아넣기. 합성 가능성 파괴
  • 프롬프트 템플릿 금지 — 도구 호출 없이 "이렇게 써줘"만 있는 것은 스킬이 아님
  • 테스트 없는 publish — 3개월 후 미안한 상황 발생
  • 타입 없는 parameters — 실수의 온상
  • 사일런트 실패 — 에러를 삼키고 성공한 척
  • 하드코딩된 경로 — /Users/alice/projects/ 같은 것
  • 문서 필드 없는 frontmatter — 라우팅이 망가짐
  • 버전 역행 — 마이너 업그레이드가 파괴적 변경

성숙한 스킬의 징후

  • 6개월 이상 유지되고 있다
  • 3명 이상이 refinement PR을 보낸 적 있다
  • hermes skills info의 success_rate가 0.9 이상
  • 다른 스킬이 이 스킬을 체이닝한다
  • 팀 신입의 첫 주 온보딩에 포함된다
